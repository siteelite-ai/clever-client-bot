

# Бета-тест новой логики до внедрения в production

## Цель

Прогнать новый pipeline (cache → CategoryMatcherLLM → прямой `category=` поиск) на реальных запросах **в изолированной edge-функции**, не трогая работающий `chat-consultant`. Сравнить с текущим поведением, убедиться что розетки находятся, а колодки/рамки — нет. Только после положительного теста — внедрять в основной pipeline.

## Что делаем

### 1. Изолированная beta edge-функция `chat-consultant-beta-search`

Отдельная функция, **не подключённая к виджету**. Делает только новый поиск товаров (без полного диалогового пайплайна, без persona, без knowledge base). Принимает на вход результат classifier'а и возвращает найденные товары + диагностику каждого шага.

```text
POST /chat-consultant-beta-search
body: { query_word: "розетка", modifiers: ["чёрный","двухместная"] }
       │
       ▼
[stage 1] getCategoriesCache() → полный список pagetitle
[stage 2] CategoryMatcherLLM → matched pagetitle[]
[stage 3] параллельный category= поиск
[stage 4] FilterLLM на схеме → options
[stage 5] финальный поиск с options
[stage 6] relaxed fallback при необходимости
       │
       ▼
{
  matched_categories: [...],
  raw_results_count: N,
  filtered_results_count: M,
  results: [...],
  timings: { cache_ms, matcher_ms, search_ms, filter_ms, total_ms },
  trace: ["cache hit", "matcher returned 3", "search returned 47", ...]
}
```

### 2. Расширение `search-products` action `list_categories`

Нужно для шага 1. Читает каталог категорий (вариант A через эндпоинт API если есть, иначе вариант B через пагинацию товаров).

### 3. Тестовый рантайм в Lovable: страница `/dev/beta-search`

Простая dev-страница (защищённая авторизацией админа), на которой:

- Поле ввода запроса в свободной форме
- Кнопка «Запустить бета-тест»
- Под капотом: вызов classifier'а (повторно используем существующий) → передача в `chat-consultant-beta-search` → отображение полного trace и списка найденных товаров
- Бок о бок: вызов того же запроса в текущий `chat-consultant` (через тот же поисковый блок, без полного диалога) и показ его результата для сравнения
- Заранее заданные **тест-кейсы** одной кнопкой «прогнать всё»:

| # | Запрос | Что проверяем |
|---|---|---|
| 1 | «нужна чёрная двухместная розетка» | основной кейс из бага: розетки да, колодки нет |
| 2 | «розетка с заземлением 16А» | морфология + технические фильтры |
| 3 | «выключатель Schneider Atlas одноклавишный» | другая категория, бренд + модификатор |
| 4 | «удлинитель на 5 розеток» | НЕ должен матчиться в категорию «Розетки» |
| 5 | «рамка для розеток Legrand белая 3 поста» | рамки именно как рамки, не розетки |
| 6 | «лампочка светодиодная E27 теплая» | категория из другого семейства |
| 7 | «провод ПВС 3х2.5» | кабельная продукция |
| 8 | «квантовый телепортер» | пустой матч → fallback → soft 404 |
| 9 | «штепсель» (синоним розетки) | проверка робастности matcher'а к синонимам |
| 10 | «двухпостовой» (только модификатор без категории) | поведение при пустой категории от classifier'а |

### 4. Сбор результатов

После прогона — таблица:

| # | Запрос | Beta: matched_categories | Beta: товаров | Beta: время | Current: товаров | Current: время | Вердикт |
|---|---|---|---|---|---|---|---|

Вердикт ставится автоматически по правилам:
- ✅ если matched_categories содержит только семантически правильные категории И товаров > 0 И время < 8с
- ⚠️ если товаров > 0, но в matched_categories попало что-то спорное
- ❌ если 0 товаров там, где current даёт > 0 (регрессия) ИЛИ в matched_categories попала явно нерелевантная категория

Результаты сохраняются в новую таблицу `beta_search_runs` для истории прогонов.

### 5. Решение о внедрении

После прогона показываю тебе таблицу. Если всё ✅ — внедряем в основной `chat-consultant` (заменяем bucket-логику на новую). Если есть ⚠️/❌ — сначала чиним промт CategoryMatcher'а или логику кеша/relaxed, прогоняем тест ещё раз, и только после зелёной таблицы внедряем в production.

## Что меняется в production до внедрения

**Ничего.** Виджет, основной `chat-consultant`, persona, search-pipeline — всё работает как сейчас. Бета-функция и dev-страница изолированы.

## Файлы

| Файл | Действие |

| `supabase/functions/search-products/index.ts` | Добавить ветку `action: "list_categories"` |

## Что НЕ трогаем

- Основной `chat-consultant` — ни строчки
- Виджет, embed.js, persona, conversational rules, knowledge base
- `prioritizeBuckets` и bucket-логику — пока бета не подтвердится
- Существующие edge-функции кроме `search-products` (только +1 action)

Деплой: автодеплой edge-функций, миграция через подтверждение пользователем.

## Ход работ

1. Миграция `beta_search_runs` (с твоим подтверждением)
2. Расширение `search-products` action `list_categories`
3. Новая edge-функция `chat-consultant-beta-search`
4. Dev-страница `/dev/beta-search`
5. Прогон 10 тест-кейсов, сохранение результатов
6. Показываю таблицу, обсуждаем
7. (Отдельным заходом, после твоего ОК) внедрение новой логики в основной `chat-consultant` и удаление bucket-логики

## Итог бета-теста, который мы хотим увидеть

Запрос «чёрная двухместная розетка»:
- `matched_categories` = розеточные подкатегории каталога, без колодок/рамок/удлинителей
- товаров > 0
- время < 5 секунд
- current `chat-consultant` на том же запросе даёт меньше товаров или мусорные категории

Если так — двигаем в production.
