
# Консилиум по аудиту: почему бот снова ошибается

Провёл аудит по релевантным ролям из `AUDIT_PROMPT.md` с опорой на код и runtime-логи:

## Роли консилиума

### 1) Edge Functions Stability Auditor
**Вердикт: не одобряет текущую реализацию**
- Логи показывают, что механизм **формально срабатывает правильно**, но путь слишком хрупкий и медленный.
- По запросу `кемпинговый` было:
  - `price_intent=most_expensive`
  - `product_category="кемпинговый фонарь"`
  - `Probe: total=6`
  - затем `Fetched all 6 of 6 products`
- Но probe занял около **9.5с**, а полный fetch ещё около **6.2с**. То есть даже когда логика сработала, она уже живёт на грани таймаутов и нестабильности.

### 2) RAG Quality Auditor
**Вердикт: не одобряет качество поиска в текущем виде**
- Главная проблема уже не в том, что бот “забыл”, что ищет самый дорогой.
- Главная проблема в том, что он ищет **по слишком буквальному запросу**: `query="кемпинговый фонарь"`.
- Каталог 220volt называет похожие товары по-разному: `прож/кемп.`, `туристический`, `ручной`, `прожектор`, сокращения и смешанные названия.
- Из-за этого бот может честно отсортировать **не весь релевантный пул**, а только тот кусок, который вернул один текстовый запрос.

### 3) Chat Conversion Auditor
**Вердикт: не одобряет поведение после ухода в сторону**
- По вашим скриншотам видно вторую проблему: когда вы увели диалог в сторону (“для охоты”), а потом вернулись к “кемпинговый”, система уже неустойчиво восстанавливает сценарий.
- Сейчас восстановление контекста держится на:
  - последних 4 сообщениях для `classifyProductName`
  - и отдельно на LLM-кандидатогенерации по истории
- Это означает: если разговор ушёл в другую ветку, текущая реализация может потерять именно **ценовой сценарий уточнения**, даже если общий контекст ещё частично помнит.

## Что именно показали логи

### Сценарий 1: “самый дорогой фонарь” → “кемпинговый”
Логи подтверждают:
- ценовой интент был распознан корректно
- уточнение после вопроса бота было распознано корректно
- система выполнила `handlePriceIntent`
- система отсортировала **6 найденных товаров** по `most_expensive`

То есть здесь проблема не “память потерялась”, а:
**она выбрала самый дорогой из 6 найденных по одному query, а не самый дорогой из всей реальной подкатегории каталога.**

### Сценарий 2: увод в сторону → потом “давай тогда кемпинговый”
Здесь корень проблемы другой:
- система не хранит явное состояние “мы находимся в режиме уточнения ценового запроса”
- она пытается восстановить всё из короткой истории
- после боковой ветки диалога это восстановление становится ненадёжным

Итог:
- иногда она отвечает как обычный подбор
- иногда вообще говорит “не найдено”
- иногда находит, но не тот экстремум по цене

## Точный root cause

```text
Проблема состоит из двух независимых дефектов:

1. Поиск экстремума цены строится на одном текстовом query
   "кемпинговый фонарь"
   вместо набора синонимов/вариантов категории.
   Поэтому пул товаров неполный.

2. Сценарий уточнения не хранится как явное состояние диалога.
   Он восстанавливается эвристически из последних сообщений.
   После смены темы это восстановление ломается.
```

## Почему текущую работу нельзя считать принятой
Потому что логика сейчас:
- иногда помнит ценовой интент
- но не гарантирует полный охват нужной подкатегории
- и не гарантирует устойчивость после ответвлений в диалоге

То есть ответ может быть:
- логически согласованным,
- но **фактически неправильным** относительно каталога.

## План исправления после аудита

### Шаг 1. Разделить две задачи: “помню интент” и “нахожу полный пул”
Не полагаться только на `recentHistory.slice(-4)`.

Нужно ввести явное состояние ценового уточнения:
- `pending_price_intent`
- `pending_category`
- `pending_need_clarification=true`

И сохранять его на 1–2 следующих хода, пока пользователь не уточнит тип.

### Шаг 2. Для price-intent использовать не один query, а multi-candidate search
Для `кемпинговый фонарь` искать не только:
- `кемпинговый фонарь`

но и варианты, которые уже умеет генерировать система:
- `фонарь`
- `фонарь кемпинговый`
- `туристический фонарь`
- `прож/кемп`
- при необходимости другие синонимы через существующий candidate pipeline

Потом:
- объединить результаты
- дедуплицировать
- только после этого сортировать по цене

### Шаг 3. Не использовать LLM-уточнение как единственный носитель состояния
Когда бот спросил:
“какой тип фонаря вас интересует?”
следующий ответ пользователя должен сначала проверяться на наличие активного `pending_price_intent`, а уже потом идти в обычную классификацию.

### Шаг 4. Сузить критерий “не найдено”
Если точный query не дал результатов:
- не говорить сразу “не найдено”
- пробовать расширенный набор синонимов категории
- и только потом уходить в fallback-ответ

### Шаг 5. Разделить ответы:
- **ценовой экстремум**: “самый дорогой / самый дешёвый”
- **рекомендационный подбор**: “какой подходит для охоты”
Сейчас эти сценарии смешиваются, из-за чего после боковой ветки цена и подбор начинают мешать друг другу.

## Что менять в коде

### `supabase/functions/chat-consultant/index.ts`

#### 1. Усилить `handlePriceIntent`
Сейчас он принимает один `query: string`.
Нужно перевести его на массив кандидатов:
- основной query
- синонимы/альтернативные формулировки
- объединённый пул результатов

#### 2. Добавить явный state для price clarification
На уровне обработки сообщений:
- если есть активное ожидание уточнения для price intent,
- текущее короткое сообщение (`кемпинговый`, `налобный`, `для охоты`) трактовать как продолжение именно этого сценария.

#### 3. Переиспользовать существующий `generateSearchCandidates`
У вас уже есть сильная логика генерации кандидатов с учётом истории.
Её нужно встроить и в ветку `price_intent`, а не искать по одному literal-query.

#### 4. Переписать ветку `not_found` для ценовых уточнений
Чтобы “кемпинговый” после уточнения не превращался сразу в:
“в каталоге ничего не найдено”,
если просто не сработал один текстовый запрос.

## Ожидаемый результат после исправления

### Будет исправлено
- “самый дорогой фонарь” → “кемпинговый” → корректный поиск по расширенному пулу
- уход в сторону разговора и возврат к уточнению перестанут ломать сценарий
- снизится число ложных ответов “не найдено”

### Что не обещает даже исправленная версия
Если API 220volt сам не умеет серверную сортировку и часть релевантных товаров не находится ни по одному разумному query, абсолютную математическую полноту гарантировать нельзя.
Но качество будет **существенно выше текущего**, потому что проблема сейчас именно в узком literal-search.

## Рекомендация консилиума
**Текущую работу действительно не стоит принимать.**

Рекомендуем новый этап реализации:
1. явное сохранение состояния ценового уточнения,
2. multi-candidate search для `price_intent`,
3. мягкий fallback вместо мгновенного `не найдено`.

Это уже не точечный фикс, а архитектурное исправление причины ошибки.
