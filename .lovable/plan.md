
Причина, почему это не сработало, по логам уже видна точно:

## Что произошло на самом деле

Запрос **не пошёл в новую category-first / bucket-N логику**, которую мы недавно чинили.

Вместо этого сработала **ветка product_search slot refinement**:

- `[Slots] product_search slot resolved...`
- `[Chat] product_search slot: refinementText="Есть черная двухместная розетка?" ...`
- `[Chat] Fetched 50 schema products for category="Розетки"`

И дальше сразу `resolveFiltersWithLLM(...)` по этим 50 товарам.

То есть в этом кейсе **новая логика с `CategoryOptionsSchema` и `prebuiltSchema` вообще не применялась**.  
Это первый ответ на вопрос “почему не сработало”.

## Но это не единственная проблема

В этом конкретном запросе дело **не только** в том, что не взяли полную схему.

По логам внутри slot-ветки видно:

- схема уже содержала нужный ключ  
  `kolichestvo_razyemov (Количество разъемов): 3, 2, 1`
- LLM вернула только:
  `{"filters":{"cvet__tүsі":"чёрный"}}`
- после валидации получилось:
  `resolved={cvet__tүsі:"чёрный//қара"}, unresolved=["черная","двухместная"]`

Здесь скрыт второй баг:

### Баг 1 — resolved есть, а modifier остаётся unresolved
Цвет успешно зарезолвился, но слово **"черная"** всё равно осталось в unresolved.

Почему: логика привязки `source_modifier` не умеет нормально считать, что:

- `черная` = `чёрный`
- `накладная` = `накладной`

Из-за этого фильтр считается найденным, но сам модификатор не помечается как “закрытый”.

Это подтверждается и старым состоянием слота:

- `existingUnresolved="накладная"`
- при этом в `existingFilters` уже есть  
  `tip_montagha__... = "накладной//..."`

То есть **"накладная" уже была резолвлена раньше, но всё равно осталась в unresolved_query**.

### Баг 2 — slot-ветка тащит unresolved слова в literal query
После этого код делает:

- `mergedFilters = existing + new`
- `mergedQuery = "накладная черная двухместная"`

И уходит в API с двумя вещами сразу:

- структурные фильтры:
  - монтаж = накладной
  - заземление = Да
  - цвет = чёрный
- плюс literal query:
  - `"накладная черная двухместная"`

Это и убивает результат:

- `[Search] query="накладная черная двухместная" → 0 results`

То есть API уже получил нормальные `options[...]`, но сверху ему ещё навязали шумный текстовый поиск по названию.

## Итого: корень проблемы — цепочка из 3 причин

### 1. Запрос ушёл не в новую ветку
Сработал `product_search slot`, поэтому фиксы с полной схемой категории не применились.

### 2. В slot-ветке сломан учёт resolved/unresolved
Слова вроде `черная/чёрный`, `накладная/накладной` не схлопываются в один смысловой модификатор.

### 3. Slot-ветка продолжает слать unresolved-текст в query
Даже когда часть фильтров уже точно резолвлена в `options[key]=value`.

## Самый важный вывод

В этом кейсе **не надо усиливать системный промпт**.  
Проблема не в том, что “LLM тупая и не знает слово”.

Проблема системная:

- запрос пошёл мимо новой ветки,
- slot state хранит мусор в unresolved,
- и потом этот мусор попадает в API query.

## Что нужно исправить дальше

1. **Протянуть full category schema в slot refinement ветку**  
   чтобы она тоже использовала `getCategoryOptionsSchema / prebuiltSchema`, а не выборку 50 товаров.

2. **Исправить attribution resolved↔modifier**  
   чтобы `черная → чёрный`, `накладная → накладной` считались уже закрытыми модификаторами и не оставались в unresolved.

3. **Добавить suppress-query в slot refinement ветку**  
   по той же логике, что уже есть в STAGE 2:
   если фильтры резолвлены, не тащить эти слова обратно в literal query.

4. **Чистить slot.unresolved_query от уже-resolved модификаторов**  
   чтобы старый мусор вроде `"накладная"` не переезжал в следующий ход.

## Коротко для отчётности

Новая логика не сработала, потому что запрос был обработан через **ветку уточнения активного product_search slot**, а не через обновлённый category-first/bucket-N pipeline.  
Внутри этой ветки остались старые дефекты: отсутствие `prebuiltSchema`, неверное определение unresolved для словоформ (`черная/чёрный`, `накладная/накладной`) и отсутствие suppress-query.  
Именно поэтому в финальный API-запрос попал шумный literal query `"накладная черная двухместная"`, который обнулил выдачу, несмотря на уже найденные структурные фильтры.
