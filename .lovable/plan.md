

# План: Исправление трекинга модификаторов в resolveFiltersWithLLM

## Суть проблемы

Всё работает правильно до строки 2358. LLM корректно определяет: «двухместные» → `kolichestvo_razyemov = "2"`, «два гнезда» → `kolichestvo_razyemov = "2"`. Фильтр валидируется. Но потом код пытается пометить исходный модификатор как «обработанный»:

```text
norm("двухместные") === norm("2")  →  false
```

Модификатор остаётся в `unresolved`. На STAGE 2 строится запрос: `options[kolichestvo_razyemov]=2 + query="черные двухместные"` — API не находит товары по тексту «черные двухместные» → 0 результатов.

Проблема **только** в трекинге (строки 2355-2361), а не в самой LLM. LLM уже работает правильно.

## Что нужно исправить

### Файл: `supabase/functions/chat-consultant/index.ts`, строки 2352-2361

Сейчас после успешной валидации фильтра код помечает модификатор обработанным только если `norm(mod) === norm(value)` или `caption.includes(norm(mod))`. Это не работает для синонимов (двухместные ≠ 2, два гнезда ≠ 2).

**Исправление**: если фильтр успешно провалидирован (ключ и значение найдены в схеме), нужно пометить ВСЕ модификаторы, которые LLM использовала для этого фильтра. Способ — после валидации каждого фильтра, проверять каждый модификатор через расширенную эвристику:

1. Прямое совпадение: `norm(mod) === norm(value)` (как сейчас)
2. Caption содержит модификатор: `caption.includes(norm(mod))` (как сейчас)  
3. **Новое**: числовое извлечение — если `value` это число, а модификатор содержит это число или его словесную форму (два→2, три→3, четыре→4, пять→5, шесть→6, одно→1, двух→2, трёх→3, четырёх→4, пяти→5)
4. **Новое**: модификатор содержит корень ключа или caption (например, «мест» в «двухместные» ↔ caption содержит «мест» или «разъём»)
5. **Новое**: обратная проверка — если LLM вернула ключ, и модификатор НЕ совпал ни с одним другим ключом, но количество модификаторов = количеству фильтров, связать оставшиеся 1-к-1

Это **одно точечное изменение** в ~15 строк. Никаких fallback-ов, никаких новых этапов. Просто правильный трекинг того, что LLM уже корректно определила.

### Деплой

После изменения — деплой edge function `chat-consultant`.

## Объём

Один файл, одна функция, одно место (~15 строк). Логика поиска не меняется — меняется только пометка обработанных модификаторов.
