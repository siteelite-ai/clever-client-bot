## Что уже видно по текущим логам

Для запроса `найди черные двухгнездые розетки` сейчас подтверждено следующее:

1. `S2` сработал правильно:
   - `search_modifiers = ["черные", "двухгнездные"]`
   - `critical_modifiers = ["двухгнездные"]`
2. `Category Resolver` тоже сработал правильно:
   - `pagetitle = "Розетки"`
3. В `facet-matcher` есть только лог `input`, но нет `result`.
4. После этого `s-search` пошёл безуспешно по `as_is_ru` и `en_translation`, оба раза `totalFromApi = 0`.

Это означает две вещи:

- Практически сейчас пайплайн НЕ делает тот «умный» семантический маппинг, который вы описали. В коде реально работает только точное сопоставление modifier ↔ value фасета. Это прямо зафиксировано в `catalog/facet-matcher.ts` и `s-search.ts`.
- Дополнительно есть слепая зона: если `facet-matcher` выходит раньше времени со статусом `no_facets` или `category_unavailable`, он вообще не пишет `result`-лог. По текущему логу это очень похоже именно на такой ранний выход.

## План исправления

### Шаг 1. Закрыть слепую зону в `facet-matcher`

Добавить явные логи для ранних выходов, чтобы различать:
- `category_unavailable`
- `no_facets`
- `no_matches`
- `ok`

Что именно логировать:
- `facetsResult.status`
- `options_count`
- `totalProducts`
- `source = cache | live | unavailable`
- `matchedModifiers` / `unmatchedModifiers`
- признак `cacheHit`

Результат: после одного повтора запроса будет точно видно, проблема в пустых facets, в кэше или в самом матчинге.

### Шаг 2. Убрать хрупкость точного матчинга

Без изменения архитектуры добавить безопасную нормализацию, которая нужна в любом случае:
- нормализация `е/ё`
- унификация регистра и пробелов уже есть, её сохранить
- тесты на русские формы вроде `черные` ↔ `чёрный`

Это низкорисковый фикс, который не нарушает data-agnostic контракт.

### Шаг 3. Реализовать реальный semantic facet resolution перед `s-search`

Сейчас этого фактически нет. Нужно довести поведение до той логики, которую вы ожидаете:

```text
user query
→ category resolved
→ category options loaded
→ modifiers classified against live facet schema
→ optionFilters built
→ s-search with category + optionFilters
```

Что будет сделано:
- добавить отдельный LLM/tool-этап для сопоставления modifiers с живыми facet options категории
- на вход давать только:
  - `pagetitle`
  - modifiers из S2
  - список реальных facets/values из `/categories/options`
- на выходе получать структурированный набор matched filters
- всё, что не удалось уверенно сопоставить, молча оставлять unmatched
- никаких хардкодов под 220volt

Это и будет тем механизмом, который сможет понимать, что:
- `черные` относится к фасету цвета
- `двухгнездные` должно маппиться на соответствующий живой value в характеристиках категории, даже если в API это формулируется не буквально так же

### Шаг 4. Сохранить текущие инварианты поиска

При внедрении semantic resolution сохранить без изменений:
- category-first архитектуру
- hard ban на `price=0`
- soft fallback с `droppedFacetCaption`
- запрет автосужения воронки вне явного фильтра
- V1 не трогать

### Шаг 5. Добавить регрессионные тесты

Покрыть тестами:
- ранний выход `facet-matcher` с отдельными логами
- `е/ё` нормализацию
- кейс `черные ... розетки` → матч по цвету
- кейс `двухгнездные ...` → матч в live-like facet schema через semantic resolver
- e2e: если facet filters собраны, то в `s-search` реально уходят `optionFilters`, а не только `query`
