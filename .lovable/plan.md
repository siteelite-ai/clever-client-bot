

# Итоговый вердикт: Edge Functions Stability Auditor (Role 4) — Финальное решение

Роль 4 отвечает за латентность и стабильность — именно она принимает итоговое решение по оптимизации скорости.

---

## Анализ двух подходов

### Подход A: Regex-эвристики (текущий + одобренные оптимизации)
- `extractTokens >= 3 || extractSpecs >= 1` — ломается на "кабель КГ 4*2,5" (2 токена, 0 спек)
- Нужно постоянно латать regex под новые форматы
- Добавление timeout, skip Pass 2, skip English fallback — всё это заплатки вокруг сломанной эвристики

### Подход B: Микро-LLM классификатор
- AI за ~1 секунду определяет: есть ли в сообщении название товара и извлекает его
- Покрывает ВСЕ форматы без regex: "Найди мне кабель КГ 4*2,5 есть в наличии" → `"кабель КГ 4*2,5"`
- При timeout (3с) — просто пропускается, идём в обычный pipeline

---

## Вердикт: ПОДХОД B ЗАМЕНЯЕТ ЧАСТЬ ОПТИМИЗАЦИЙ

### Что становится НЕ НУЖНО при микро-LLM:

1. ~~Расширение `extractSpecs` для сечений кабеля~~ — AI сам поймёт формат
2. ~~Смягчение порога токенов (2 вместо 3 с числами)~~ — AI решает, а не regex
3. ~~Исправление `cleanQueryForDirectSearch` для "есть в наличии"~~ — AI сам очистит название

Эти три пункта были нужны только для того, чтобы regex-эвристика корректно определяла "это название". Микро-LLM делает это лучше и универсальнее.

### Что ВСЁ ЕЩЁ НУЖНО:

1. **AbortController timeout 10с** в `searchProductsByCandidate` — защита от зависания API 220volt. Это не связано с классификацией, это защита сетевого вызова
2. **Skip English fallback при >= 1 результате** — экономия 7-10с. Не связано с классификацией
3. **Skip Pass 2 при `hasHumanFilters === false`** — экономия 7-10с. Не связано с классификацией

Эти три оптимизации относятся к **fallback pipeline** (когда title-first не сработал) и нужны независимо от способа классификации.

## Итоговый план внедрения

### Шаг 1: Микро-LLM классификатор (заменяет title-first эвристику)
- Новая функция `classifyProductName(message)` — лёгкий LLM-вызов (~1с)
- Промпт: 10-15 строк, JSON-ответ без tool calling
- Timeout: 3 секунды, при неуспехе — обычный pipeline
- Модель: `google/gemini-2.5-flash-lite` (самая быстрая и дешёвая)

### Шаг 2: Оптимизации fallback pipeline (независимо от классификатора)
- AbortController timeout 10с в `searchProductsByCandidate`
- Skip English fallback при >= 1 результате
- Skip Pass 2 при отсутствии фильтров по характеристикам

### Файл для изменения
`supabase/functions/chat-consultant/index.ts`:
- Строки 1921-1954: заменить regex-блок title-first на вызов `classifyProductName`
- Новая функция `classifyProductName` (~30 строк)
- `searchProductsByCandidate`: добавить AbortController
- Блок English fallback: изменить порог
- `searchProductsMulti`: условный skip Pass 2

### Ожидаемое время ответа
- Запрос по названию (title-first success): **~2-3с** (классификация 1с + API 7-10с + short-circuit) = **8-12с**
- Запрос по названию (title-first fail → fallback): **15-20с** вместо 40с
- Общий вопрос ("какие розетки?"): классификация 1с → `has_product_name=false` → обычный pipeline без задержки

### Риски
- **Низкий**: микро-LLM иногда ошибётся — fallback страхует
- **Нулевой**: article-first, site-id не затрагиваются
- **Минимальный**: +1с на классификацию для общих запросов — компенсируется экономией на названиях
