# Чистый вывод: меньше «технических объяснений», цена + наличие на складах, никаких товаров под заказ

Три правила, которые применяются **везде** в выдаче (основная, Soft Fallback, replacement, cross-sell, артикул):

1. Никакой «лишней технической болтовни» Composer'а про то, чего в фасете нет.
2. Карточка товара — красивый буллет-блок: ссылка → цена → бренд → наличие по складам.
3. Товары с `price = 0` («под заказ» / «по запросу») **никогда** не показываются пользователю.

## Что меняется

### 1. Composer перестаёт объяснять, чего нет (§11.2a)

Сейчас при «лампа кукуруза» (и подобных) Composer обязательно дописывает фразы вида «не нашёл в характеристиках *кукуруза*», «доступные значения для *Форма колбы*: …». Это лишний шум.

Новые правила в §11.2a:
- **`unresolved_traits`** больше **не** выводится отдельной строкой. Остаётся только в trace для аналитики.
- **`soft_matches`** по умолчанию не выводится. Строка «Ближайшее по *{trait}*: *{value}*» добавляется **только** если в выдаче ≤2 товаров (когда пользователю реально важно понимать, почему мало).
- **`applied_aliases`** по умолчанию не выводится. Строка «Учёл синонимы» появляется только если `confidence < 0.85` или 0 товаров.
- **Soft Fallback (`pending_clarification` + `applied_filters ≥ 2`)** — больше **не** пишет «*{trait}* не нашёл, показал по остальным критериям» и **не** перечисляет доступные значения. Вместо этого — **одна короткая строка в конце** ответа: «Если важно уточнить *{facet_caption}* — напишите.»
- **Блокирующий clarification** (когда товаров не показываем) — один короткий вопрос «Уточните, пожалуйста: какой *{facet_caption}*?». Список значений добавляется **только** если их ≤6 и они короткие.

Эффект на «лампа кукуруза»: вместо абзаца про отсутствие «кукуруза» в форме колбы — просто товары + при необходимости одна короткая строка.

### 2. Новый формат карточки товара (§17.3, §17.5)

Вместо одной строки `**[Name](URL)** — *price* ₸, brand` — **многострочный буллет-блок**:

```
- **[Розетка белая 16А](https://220volt.kz/p/rozetka-belaya-16a)**
  - Цена: *990* ₸
  - Бренд: Schneider
  - Наличие: В наличии — *Алматы 24 шт*, *Астана 8 шт*
```

Правила:
- Цена — *курсив*, пробелы как разделители тысяч, `₸` через пробел.
- Бренд — из `vendor`. Если пусто — строка «Бренд» **пропускается** целиком (никаких «—»).
- Наличие — из `warehouses[]` по правилам §17.5:
  - есть склад в городе пользователя → `Наличие: В наличии — *{город_пользователя} N шт*` + до 3 других;
  - в его городе нет, но есть другие → `Наличие: В вашем городе ({город}) нет на складе. Ближайший: *{город2} N шт*`;
  - город неизвестен → `Наличие: В наличии — *{город1} N шт*, *{город2} N шт*` (до 3);
  - все склады с 0 или `warehouses` пуст → строка «Наличие» **не выводится** (пустые поля никогда не показываем).
- В URL круглые скобки → `%28`/`%29`. Никакого экранирования слэшами.

### 3. Жёсткий запрет на товары `price = 0` (§9C.3)

Сейчас есть конфиг `zero_price_policy` с режимами `hide`/`show_with_marker`/`allow`. **Удаляем** — оставляем только жёсткое правило:

- `price === 0` **никогда** не попадает в вывод: ни в основной список, ни в Soft Fallback, ни в cross-sell, ни в replacement, ни в «похожие», ни в карточку артикула.
- Если после фильтрации `results = []`, но `total > 0` — Composer **не показывает товары вовсе** и выдаёт `CONTACT_MANAGER`: «По вашему запросу есть позиции, но их цена сейчас уточняется. Свяжитесь с менеджером.»
- Catalog Search догружает страницы (до `app_settings.zero_price_max_pages`, дефолт 5), пока не наберёт лимит ненулевых.
- Двойная защита: фильтр `price > 0` применяется и в Catalog Search, и непосредственно перед рендером Composer'а. Утечка → defect `zero_price_leak` (метрика обязана быть 0).
- Конфиг `zero_price_policy` **снят**; добавлен `zero_price_max_pages`.

## Чего НЕ будет

- никаких хардкодов под конкретные слова («кукуруза», «чёрный»);
- никаких «—», «н/д», «уточняется» в карточке;
- никакого режима показа товаров с 0 ₸ «с маркером» — он удаляется полностью;
- никаких длинных объяснений «не нашёл то-то, доступно то-то» в обычной выдаче.
