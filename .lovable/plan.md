## Аудит плана (3 роли)

Перед утверждением — независимый разбор силами трёх ролей. Основано на чтении реального кода (`chat-consultant/index.ts` 6271 строка, `search-products/index.ts` 197 строк) и swagger.json от 220volt.

### 1. Главный архитектор — общая реализуемость

**Вердикт: реализуемо без слома смежных процессов. Риск НИЗКИЙ.**

Аргументы:
- Новый эндпоинт `/categories/options` отдаёт ровно ту структуру (`options[].values[].{value_ru, value_kz, products_count}`), которую внутри `chat-consultant` уже строит вручную из сэмпла товаров (см. `getCategoryOptionsSchema`, строки 1286–1359). То есть мы **подменяем источник данных, не меняя контракт** — `Map<key, {caption, values:Set}>`.
- Функция `resolveFiltersWithLLM` (строка 2825) уже спроектирована с параметром `prebuiltSchema?: Map<...>`. Это и есть штатный «шов» для подмены источника — никакой инвазивной хирургии не требуется.
- Весь остальной поиск (`searchProductsByCandidate`, строка 3212) уже использует формат `options[<key>][]=<value>` — он совпадает с тем, что задокументирован в swagger. Менять не нужно.
- Категории-бакеты, классификатор неоднозначности (`classifyCategoryAmbiguity`), DISTINCT/SYNONYMS-логика, `matchCategoriesWithLLM`, кэш категорий, knowledge base, виджет — всё это **не трогаем**. Изменения локализованы в 2 функциях из ~70.

Риски и митигации:
- **R1 (средний):** новый эндпоинт может вернуть пустой `options:[]` для редкой/пустой категории → FilterLLM получит пустую схему → не сможет резолвить → текущее поведение тоже выдаст пустую схему («No options found in products, skipping»). Вывод: **деградация эквивалентна текущей**, ничего не ломаем.
- **R2 (низкий):** ключ `cvet__tүs` может в каких-то категориях писаться как `cvet__tүsі` (хвост `і`). Запрос `/products?options[cvet__tүsі][]=...` идёт строго по тому ключу, который пришёл от `/categories/options`, поэтому несоответствий быть не может — мы не хардкодим ключ.
- **R3 (низкий):** товарные карточки в `product.options[]` приходят с тем же ключом — проверено в коде (строки 1332–1340, 2702–2713). Значит scoring-логика в `searchProductsByCandidate` (строки 3359–3370, сравнение `opt.key === key`) продолжит работать.

### 2. Бэкенд edge-функций — точки касания и совместимость

**Вердикт: модификация чисто аддитивная для `search-products`, замена-под-ключ для `chat-consultant`.**

Точки изменений (всего 4 функции, ~150 строк):

| Файл | Что меняем | Риск |
|------|-----------|------|
| `search-products/index.ts` | + новый action `category_facets` | Нулевой, аддитивно |
| `chat-consultant/index.ts` → `getCategoryOptionsSchema` (строки 1286–1359) | Заменить тело: вместо «качаем 5 страниц по 200 товаров и собираем options руками» → один запрос к `/categories/options`. Сигнатура и тип возврата НЕ меняются | Низкий |
| `chat-consultant/index.ts` → `getUnionCategoryOptionsSchema` (строка 1362) | Вызывает новую `getCategoryOptionsSchema` параллельно для N категорий и сливает Map'ы. Логика слияния остаётся прежней | Низкий |
| `chat-consultant/index.ts` → `resolveFiltersWithLLM` (строка 2825) | НЕ трогаем. Уже принимает `prebuiltSchema` | — |

Обратная совместимость:
- TTL-кэш и сигнатура `{ schema, productCount, cacheHit }` сохраняются. `productCount` начинаем брать из `category.total_products` нового ответа — это даже точнее (раньше считали ≤1000, теперь все 2078).
- 6 мест вызова `resolveFiltersWithLLM` в коде (строки 3353, 4123, 4490, 4693, 4766, 4979, 5146, 5204) автоматически получают новый источник — без правок.
- Если `/categories/options` вернёт 404/таймаут → fallback на старую логику (сэмплинг). То есть откатиться можно фича-флагом за 1 строку.

Снижение нагрузки:
- Сейчас на каждый «холодный» запрос вида «розетки чёрные» — до 5 параллельных запросов по 200 товаров = ~1000 объектов JSON ≈ 2-5 МБ.
- После: 1 запрос ~3 МБ структурированной схемы (для «Розеток» — 2.9 МБ, проверено), но **без скачивания самих товаров для построения схемы**. Поиск товаров делается отдельным целевым запросом уже с применёнными `options[...]=...`, страница 1, per_page 12 — несколько килобайт.
- Итог: −80% трафика к API магазина на типовом запросе.

### 3. AI/Prompt-инженер — поведение FilterLLM и обработка цвета

**Вердикт: качество резолвинга вырастет, два цветовых ключа учитываются естественно, доп.промптов не требуется.**

- FilterLLM получает строки вида `KEY="cvet__tүs" | Цвет | values: белый, чёрный, графит, ...`. Сейчас при сэмплинге из 30 товаров часть значений обрезается → LLM физически не может вернуть «графит», если в выборку он не попал. После — все 61 значение цвета на «Розетках» доступны.
- **Два «цветовых» ключа** автоматически попадают в схему, если они есть в категории:
  - `cvet__tүs` (или `cvet__tүsі`) — Цвет корпуса
  - `cvetovaya_temperatura__k__tүs_temperaturasy__k` — Цветовая температура
  
  LLM сама выбирает подходящий по семантике модификатора («тёплый» → температура, «чёрный» → цвет корпуса). Никакого хардкода. Если категория «Лампы» и модификатор «белый» — LLM вернёт оба варианта, и обработчик `unresolved` запросит уточнение через quickReplies (это уже есть в текущем коде, строки 1697–1731).
- `value_kz` теперь приходит отдельным полем — можно строить bilingual values для матча: `${value_ru}//${value_kz}`. Это совпадает с форматом, который уже используется в коде (`opt.value.split('//')`).

Единственное правило в системном промпте FilterLLM **не меняем**: «выбирай ключи только из переданной схемы». Этого достаточно.

## План реализации (после аудита)

### Шаг 1. `search-products`: добавить action `category_facets`

В `supabase/functions/search-products/index.ts` добавить ветку:

```ts
if (body?.action === 'category_facets') {
  const { pagetitle, categoryId } = body;
  const url = categoryId
    ? `https://220volt.kz/api/categories/${categoryId}/options`
    : `https://220volt.kz/api/categories/options?pagetitle=${encodeURIComponent(pagetitle)}`;
  // fetch с Bearer-токеном, кэш в Map TTL 1ч по ключу `${pagetitle||id}`,
  // возвращаем { category, options } (аккуратно разворачиваем двойной data.data)
}
```

Существующие ветки (`list_categories`, поиск товаров) **не трогаем**.

### Шаг 2. `chat-consultant`: подменить источник схемы

Внутри `getCategoryOptionsSchema(categoryPagetitle, apiToken)`:
- Вместо цикла «5 страниц × 200 товаров» — один `fetch` к `/categories/options?pagetitle=...`.
- Преобразовать ответ в существующий тип `Map<string, {caption: string, values: Set<string>}>`:
  - `caption` = `caption_ru` (формат как раньше, без `//`)
  - `values` = Set из `${value_ru}//${value_kz}` (совместимо с парсингом `.split('//')[0]` ниже по коду)
- `productCount` = `category.total_products`.
- Кэш и сигнатура без изменений.
- Применить `isExcludedOption(key)` (уже есть фильтр служебных ключей типа `fayl`, `kodnomenklatury`).

`getUnionCategoryOptionsSchema` остаётся как есть — она просто параллельно зовёт новую `getCategoryOptionsSchema` и сливает Map'ы.

### Шаг 3. Fallback и фича-флаг

В новой `getCategoryOptionsSchema`:
- При HTTP-ошибке или пустом `options:[]` логируем `[CategoryOptionsSchema] facets API failed → falling back to product sampling` и вызываем старую реализацию (переименуем её в `getCategoryOptionsSchemaLegacy`).
- Это страховка на первые 1-2 дня после деплоя; если в логах за неделю ноль вызовов legacy — удалим.

### Шаг 4. Два цветовых ключа

В системном промпте FilterLLM (строки ~2900–2920) — **никаких изменений**. Промпт уже говорит «выбери из переданных KEY». Когда схема содержит оба ключа, LLM сама их различит. Добавим только подсказку в комментарий схемы для модели:
```
Если модификатор про цвет корпуса (чёрный, белый, графит) — ищи KEY с caption "Цвет".
Если модификатор про оттенок света (тёплый, холодный, 4000K) — ищи KEY с caption "Цветовая температура".
```
Одна строка в промпте, не больше.

### Шаг 5. Память и документация

Обновить `mem://features/search-pipeline`: новый источник схемы — `/categories/options`, fallback на сэмплинг отключается через 1 неделю.

## Что НЕ делаем (зафиксировано аудитом)

- Не переписываем `resolveFiltersWithLLM`.
- Не трогаем `searchProductsByCandidate`, scoring, slot-state, классификаторы DISTINCT/SYNONYMS.
- Не трогаем кэш категорий (`getCategoriesCache`), он питает другой кодпуть.
- Не меняем UI чата, виджет, knowledge base, аналитику.
- Не хардкодим словари цветов, не делаем парсинг HTML каталога.

## Критерии приёмки

1. В логах `chat-consultant` после деплоя на запрос «розетки чёрные» появляется строка `[CategoryOptionsSchema] /categories/options HIT for "Розетки": 53 keys, 2078 products, …ms` (вместо нынешней `… from N products …`).
2. Время первого ответа на холодный запрос «розетки чёрные» < 2.5 с (было 4-7 с из-за 5×200).
3. Запрос «лампа тёплый свет» резолвит `cvetovaya_temperatura__k__tүs_temperaturasy__k`, запрос «лампа чёрная» — `cvet__tүs`.
4. Старые сценарии (поиск по бренду, цене, артикулу, категории-бакеты, ambiguity-классификатор) проходят без регрессий — проверим вручную 5-7 типовых сценариев после деплоя.
5. В edge-логах нет ни одного вызова `getCategoryOptionsSchemaLegacy` за 7 дней → удаляем legacy-код.
