

## Аудит фичи «Cross-sell в каталожном поиске» — 4 роли

Фича: добавить контекстный cross-sell и мягкий уточняющий вопрос в стандартный каталожный поиск (ветка `else if (productContext)`, строки 2192-2202), по аналогии с тем, что уже работает в article-first.

---

### 1. 💰 Sales Logic Auditor (Роль 2)

**Текущее состояние:** Ветка каталожного поиска содержит ТОЛЬКО правила копирования ссылок. Бот отдаёт список товаров и замолкает — нет CTA, нет cross-sell, нет мотивации к покупке.

**Риски:**
- **Главный риск — cross-sell на этапе УТОЧНЕНИЯ.** В отличие от article-first, каталожный поиск часто возвращает несколько вариантов, и LLM 2 должна задать уточняющий вопрос (серия, мощность, количество полюсов). Если добавить cross-sell безусловно — бот будет предлагать аксессуары ДО того, как клиент выбрал основной товар. Это сбивает с толку и удлиняет воронку.
- **Ложный cross-sell при широких запросах.** Запрос «перфораторы» вернёт 10 моделей — cross-sell тут неуместен, нужен выбор.

**Рекомендация:**
Cross-sell добавлять **условно** — только когда LLM 2 уже показывает конкретный товар или короткий список (1-3 товара). Инструкция должна явно запрещать cross-sell при уточняющих вопросах.

---

### 2. ⚡ Edge Functions Stability Auditor (Роль 4)

**Находки:**
- Изменение только в тексте промпта — без изменения логики, API-вызовов или типов данных. Риск поломки **минимальный**.
- Увеличение длины промпта на ~200 токенов — незначительное влияние на время ответа и стоимость.
- Warehouses уже передаются в `productContext` (реализовано ранее) — дополнительных изменений не нужно.

**Риски:** Практически отсутствуют. Это текстовое изменение в одной строковой переменной.

**Рекомендация:** Безопасно к реализации.

---

### 3. 📊 Chat Conversion Auditor (Роль 5)

**Текущая воронка (каталожный поиск):**
1. Пользователь: запрос → LLM 1 извлекает параметры → API → LLM 2 показывает товары
2. Если нужно уточнение → ещё 1-2 шага
3. Товар выбран → **мёртвая точка** — бот не предлагает ничего дальше

**Оптимальная воронка:**
1. ...тот же путь до выбора товара...
2. Товар показан → cross-sell: «Для монтажа понадобится DIN-рейка — подобрать?»
3. Клиент либо берёт допы, либо отвечает → «Что ещё подобрать для вашего проекта?»

**Ключевое правило:** Cross-sell работает на конверсию ТОЛЬКО после выбора. До выбора он мешает.

**Рекомендация:** Инструкция должна содержать **явное условие**: «Если ты задаёшь уточняющий вопрос по выбору — НЕ добавляй cross-sell. Cross-sell добавляй ТОЛЬКО когда показываешь конкретный выбранный товар или рекомендацию.»

---

### 4. 🎨 Widget UX Auditor (Роль 7)

**Находки:**
- Markdown рендерится корректно — cross-sell в формате текста не создаст проблем с отображением
- Длина ответа увеличится на 1-2 предложения — приемлемо

**Риск:** При каталожном поиске ответ уже может быть длинным (5-10 товаров + характеристики). Добавление cross-sell к и без того объёмному ответу может перегрузить экран виджета.

**Рекомендация:** Cross-sell — максимум 1-2 предложения. Не список, не буллеты, а одна фраза: «Для монтажа также понадобится DIN-рейка — могу подобрать?»
