
# План: ускорить поиск по названию и не ломать текущую логику

## Что показывает аудит

Проблема сейчас двойная:

1. Поиск по названию слишком медленный
   - запрос проходит через несколько слоёв: direct search → LLM 1 → `searchProductsMulti` → Pass 1 → Pass 2 → English fallback → финальный LLM
   - по логам это даёт много API-вызовов и большой промпт на финальном ответе
   - особенно дорого то, что `searchProductsMulti` ограничивает Pass 1 до 3 кандидатов, но Pass 2 уже не ограничивает, а English fallback добавляет ещё серию вызовов

2. Поиск по названию иногда не находит нужный товар
   - сейчас точное/почти точное название не имеет отдельного «приоритетного» режима как у артикула
   - `query` ищется через набор кандидатных фраз, но `per_page=30` и сортировка API могут не вернуть нужный товар в первых результатах
   - потом модель видит много нерелевантных товаров и делает неверный вывод «товара нет»
   - на скриншоте это именно и произошло: в результатах были похожие светильники, а не сама лампа

## Что предлагаю сделать

### 1. Добавить отдельный быстрый режим для поиска по названию
Перед LLM и до сложного multi-search:

- нормализовать пользовательский текст
- вытащить «ядро названия» без мусорных слов вроде:
  - «есть»
  - «в наличии»
  - «сколько стоит»
  - «подскажите»
- запустить 2 параллельных title-first запроса:
  - `query = полное очищенное название`
  - `query = укороченная ключевая версия названия`

Если один из них возвращает хороший матч, сразу short-circuit, как у артикула.

Идея: для названий нужен такой же приоритетный путь, как для article/site id, только мягкий, не exact-match.

### 2. Ввести серверную проверку «хорошего совпадения»
Не просто «есть какие-то товары», а именно оценка, насколько результат похож на запрос.

Добавить score на сервере:
- совпадение бренда
- совпадение модельного кода (`T8`, `G13`, `MR16`)
- совпадение числовых характеристик (`18Вт`, `6500К`, `230В`)
- доля слов из названия пользователя, найденных в `pagetitle`

Если score высокий:
- возвращаем товар сразу
- пропускаем LLM 1 и двухпроходный поиск

Если score низкий:
- продолжаем обычный pipeline

Это безопасно, потому что не заменяет старую логику, а только добавляет быстрый выход при явном совпадении.

### 3. Ограничить дорогие ветки поиска
Сейчас latency раздувается из-за каскада fallback-логики. Нужно ужесточить бюджет:

- ограничить Pass 2 тем же cap, что и Pass 1
- English fallback запускать только если:
  - найдено 0 результатов, либо
  - найденные результаты имеют очень низкий title-score
- не запускать English fallback для уже русско-английских брендовых/модельных запросов типа `ECO T8 18Вт 6500К G13`
- уменьшить шумные кандидаты вроде `led`, `lamp`, `linear`, если у нас уже есть сильный query из исходного сообщения

### 4. Поднять приоритет «полного названия» в генерации кандидатов
В `extractionPrompt` уточнить:

- если пользователь ввёл полное или почти полное название товара, первый кандидат должен быть максимально близок к исходному названию
- не дробить такой запрос слишком агрессивно на общие слова
- модельные коды и ключевые характеристики сохранять в первом кандидате обязательно

Это поможет LLM не превращать точный пользовательский ввод в слишком общие поисковые фразы.

### 5. Улучшить финальную фильтрацию перед ответом «товар не найден»
Перед тем как говорить, что товара нет:

- если есть хоть какие-то результаты, прогнать локальный rerank по title-score
- если находится товар с высоким совпадением по названию/характеристикам — показать его
- только если score действительно низкий у всех кандидатов, разрешать ответ «не найдено»

Это уберёт системную ошибку, когда товар есть глубже в выдаче или среди похожих результатов, но бот преждевременно отказывается.

## Технические изменения

### Файл
- `supabase/functions/chat-consultant/index.ts`

### Что именно менять
1. Добавить helpers:
   - нормализация title-query
   - извлечение токенов/характеристик
   - scoring/rerank результатов
2. Расширить блок до LLM:
   - article/site id short-circuit оставить как есть
   - после него добавить `name-first short-circuit`
3. Обновить `searchProductsMulti`:
   - cap и для Pass 2
   - более строгие условия для English fallback
4. Обновить `generateSearchCandidates` / `extractionPrompt`
5. Перед формированием `productContext` делать rerank найденных товаров

## Почему это безопасно

- article-first остаётся первым и не ломается
- site-id fallback остаётся как есть
- старая LLM-логика не удаляется, а используется только когда быстрый путь не дал уверенного результата
- изменения локализованы в одной edge function
- риск регрессии низкий, потому что добавляется приоритетный short-circuit, а не полный переписанный поиск

## Ожидаемый результат

После внедрения:
- точное или почти точное название должно находиться заметно быстрее
- бот не должен уходить в длинный каскад поиска, если уже есть хороший title-match
- уменьшится количество ложных ответов «товар не найден», когда товар реально есть
- общая логика по артикулам, брендам, фильтрам и базе знаний сохранится

## Порядок внедрения

1. Добавить title-first short-circuit с scoring
2. Ограничить Pass 2 и English fallback по latency budget
3. Обновить prompt генерации кандидатов
4. Добавить защиту перед финальным ответом «не найдено»
5. После внедрения проверить сценарии:
   - точное полное название
   - частичное название
   - артикул
   - site id
   - бренд + категория
   - запросы, где товара действительно нет
