

# План: Упростить Pass 1 до одного базового запроса категории

## Идея

Вы правы — раз `resolveFiltersWithLLM` берёт на себя всю фильтрацию по модификаторам, незачем генерировать 6+ комбинаций запросов. Достаточно одного базового запроса категории (например, `"розетка"`) для Pass 1, чтобы получить широкую выборку товаров и их схему характеристик. Дальше LLM-резолвер сделает своё дело.

Это решает проблему с Гармонией: сейчас все 6 слотов заняты комбинациями `"розетка Гармония"` / `"Гармония розетка"`, API не находит ничего → Pass 1 пуст → фильтрация невозможна.

## Что меняем

**Файл**: `supabase/functions/chat-consultant/index.ts`, строки 3121–3142

Вместо генерации множества комбинаций и Cap=6 — отправляем **один запрос** с `effectiveCategory` (базовый тип товара):

```
"розетка" → Pass 1 → 30+ товаров → resolveFiltersWithLLM → Pass 2 с фильтрами
```

Конкретно:
- Убираем цикл генерации `orderedQueries` с комбинациями modifier+category (строки 3121-3136)
- Формируем один кандидат: `{ query: effectiveCategory }` 
- Pass 1 Cap больше не нужен — один запрос
- `generateCategorySynonyms` тоже больше не нужен для этого пути — можно убрать вызов (строка 3119)
- Pass 2 тоже упрощается: один запрос с resolved фильтрами вместо 3 кандидатов

## Что НЕ трогаем

- `resolveFiltersWithLLM` — основной механизм фильтрации, работает
- Промпт классификатора — работает (category="розетка", modifiers=["Гармония"])  
- Title-first, price-intent, replacement ветки
- `generateCategorySynonyms` — функцию не удаляем, просто не вызываем в category-first

## Ожидаемый результат

- **«розетки из коллекции Гармония»**: Pass 1 `"розетка"` → 30+ розеток → LLM видит `Коллекция: ГАРМОНИЯ` → Pass 2 фильтрует
- **«черная двухместная розетка»**: Pass 1 `"розетка"` → 30+ розеток → LLM видит цвет + количество мест → Pass 2 фильтрует
- **«удлинитель 50 м»**: Pass 1 `"удлинитель"` → удлинители → LLM видит длину → Pass 2 фильтрует
