

# Фикс: серверный classifier + быстрый cache + честный current-baseline

## Диагноз первого прогона

| Что увидели | Причина |
|---|---|
| `matched_categories: []` для «чёрная двухместная розетка» | Клиентский heuristic-classifier выбрал первое значащее слово — прилагательное `чёрная`. Matcher искал категорию «чёрная», её нет → пусто |
| `cache: 146484 ms` | Холодный старт: пагинация всего каталога через `search-products?action=list_categories` идёт по страницам товаров |
| `Current: 0 товаров, 7221 мс` | `search-products` дернули с сырым `query="нужна чёрная двухместная розетка"` — API 220volt на такую фразу не отвечает; baseline нечестный |

Итого: бета-тест не проверил гипотезу, потому что три внешних к ней узла были сломаны.

## Что чиним

### 1. Серверный ClassifierLLM (новая edge-функция `query-classifier`)

Изолированная микро-функция, один LLM-вызов. На вход — сырая фраза пользователя. На выход — `{ query_word, modifiers }`.

```text
"нужна чёрная двухместная розетка"
        │  ClassifierLLM (gemini-2.5-flash, structured tool call)
        ▼
{ query_word: "розетка", modifiers: ["чёрная", "двухместная"] }
```

Системный промт (без примеров категорий, по принципу пользователя):

> Извлеки из запроса покупателя электротоваров одно нормативное существительное товара (`query_word`) и список модификаторов (`modifiers`). 
> 
> Правила:
> 1. `query_word` — именительный падеж, единственное число, нижний регистр. Это **сам предмет**, который ищет покупатель, не его свойство и не сопутствующий товар.
> 2. `modifiers` — все остальные смысловые слова: цвета, материалы, бренды, технические параметры, количество. В именительном падеже, нижнем регистре. Без стоп-слов («нужна», «хочу», «для» и т.п.).
> 3. Если в запросе нет существительного-товара — `query_word` = пустая строка, всё уходит в `modifiers`.
> 4. Если запрос состоит из нескольких товаров — бери основной (последний по смыслу).

Возвращает structured tool call `extract({ query_word: string, modifiers: string[] })`.

Используется и в `BetaSearch.tsx` (вместо клиентской эвристики), и **в будущем** в основном `chat-consultant` после миграции. Изолированно, переиспользуемо, тестируемо.

### 2. Ускорение `list_categories` cache

Текущий путь — пагинация товаров (вариант B). 146с — недопустимо даже на холодном старте. Меняем стратегию:

- **Вариант A (приоритет)**: проверить наличие у API 220volt прямого эндпоинта категорий. Часто такие API экспонируют `/api/categories` или включают список в `/api/products?per_page=1` как facets/aggregations. Если есть — один запрос, ~200мс.
- **Если эндпоинта нет**: 
  - Поднять `per_page` до максимума, который API принимает (часто 200/500), и ограничить пагинацию **первыми N страницами** (например, 5–10) — для сбора `pagetitle` категорий не нужен весь каталог, категории повторяются у тысяч товаров и выходят на плато за первые ~500 SKU.
  - Параллелить страницы (`Promise.all` для page 1..10), а не последовательно.
  - Сохранять кеш в **Supabase таблицу** `category_cache` (id=1, jsonb categories, fetched_at). Тогда холодный старт edge-инстанса = 1 SELECT (~50мс), а не 146с. Перестроение раз в час по TTL — асинхронно, не блокируя запрос (если кеш просрочен — отдать старый и в фоне обновить).

### 3. Честный current-baseline

Сейчас `runCurrent` дёргает `search-products` напрямую с сырым запросом. Это **не** то, что делает production `chat-consultant` — там есть свой classifier и своя пред-обработка. Чтобы сравнение было честным:

- Использовать тот же `query_word` от ClassifierLLM (а не сырую фразу) и для current тоже: `search-products?query=розетка` — это симметрично тому, что делает текущий продакшн в текстовой ветке.
- Это покажет реальную дельту: «matcher + category=» vs «query=» при одинаковом входе.

### 4. UI бета-страницы

- Показывать на карточке результата: исходную фразу, **результат classifier'а** (`query_word` + `modifiers`), отдельной строкой. Сейчас classifier невидим, и непонятно, виноват он или matcher.
- Кнопка «Прогреть кеш» — вызывает `list_categories` форсированно, чтобы первый прогон не упирался в холодный старт.

## Pipeline после фикса

```text
сырая фраза
   │
   ▼
ClassifierLLM (новая edge-функция)
   │
   ▼
{ query_word: "розетка", modifiers: ["чёрная","двухместная"] }
   │
   ├──► [BETA]  getCategoriesCache (быстро) → matcher → category= search → FilterLLM → relaxed
   │
   └──► [CURRENT baseline] search-products?query=розетка   (тот же word, текстовый поиск)
   │
   ▼
сравнение
```
