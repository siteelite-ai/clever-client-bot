

# Ответ обрывается на полуслове: нет `max_tokens` в главном стриме + нет логирования finish_reason

## Что показал код

В `supabase/functions/chat-consultant/index.ts` строки 4594-4600 — главный вызов LLM для пользовательского ответа:

```js
{
  model: aiConfig.model,           // gemini-2.5-pro
  messages: messagesForAI,
  stream: true,
  ...DETERMINISTIC_SAMPLING,        // top_k=1, seed=42, provider
  reasoning: { exclude: true },
  // ❌ max_tokens НЕТ
}
```

Сравните с другими вызовами:
- Intent extractor (стр. 707): `max_tokens: 300` ✅
- CategorySynonyms (стр. 974): `max_tokens: 150` ✅
- FilterLLM resolver (стр. 2244): `max_tokens: 500` ✅
- **Main stream (стр. 4598): `max_tokens` НЕ ПЕРЕДАЁТСЯ** ❌

## Почему это ломает ответ

OpenRouter для Gemini-2.5-pro без явного `max_tokens` использует **дефолт провайдера** (Google AI Studio), а это часто 1024-2048 completion-токенов. Параметр `reasoning: {exclude: true}` для pro-модели **исключает reasoning из ответа клиенту, но не отключает его на стороне модели** — pro всё равно тратит часть бюджета на скрытые «мысли». В итоге на полезный текст остаётся 200-400 токенов, что и видно на скриншоте: ответ оборвался ровно после первого товара на «...очень близкий к чёрному.»

Дополнительный фактор: в стриме нет логирования финального `finish_reason`. Если бы он логировался, мы бы сразу увидели `length` (= обрыв по лимиту) вместо `stop` (нормальное завершение). Сейчас обрыв тихий.

## Фикс (минимальный, точечный)

### Шаг 1. Добавить `max_tokens` в главный стрим

Строка 4594-4600 — добавить:

```js
max_tokens: 4096,
```

4096 — безопасный потолок: средний наш ответ 800-1500 токенов, при списке из 5-7 товаров с описаниями уходит ~2500-3000. Запас 4096 закрывает все реальные сценарии и при этом не даёт модели «писать роман». Если ответ короче — `max_tokens` ничего не стоит, потому что биллинг идёт по фактическим токенам.

### Шаг 2. Логировать finish_reason в стриме

В обоих стрим-обработчиках (greeting-strip ~4705 и standard ~4807) перед `controller.close()` парсить из последнего чанка `finish_reason` и логировать:

```
[Chat] Stream finished: finish_reason=length contentLen=487  ← обрыв
[Chat] Stream finished: finish_reason=stop contentLen=1234   ← норма
```

Это даёт мгновенную диагностику: если в логах `finish_reason=length` — нужно ещё повышать лимит; если `stop` — проблема в чём-то другом (например модель сама решила остановиться).

### Шаг 3. Признать это паттерном — ВСЕ LLM-вызовы должны иметь `max_tokens`

Проверка показала: 3 из 4 вызовов уже имеют лимит, главный стрим — единственное упущение. Это нормально, но стоит закомментировать в коде причину выбора 4096, чтобы при будущих правках никто не убрал параметр «как лишний».

## Что НЕ делаем

- Не трогаем sampling-параметры, модель, pipeline, резолвер, RAG, persona, widget
- Не повышаем `max_tokens` до 8192/16384 — лишний риск, что модель будет растягивать ответ; 4096 достаточно с запасом x2-3 от реальных размеров
- Не убираем `reasoning: {exclude: true}` — он правильный, просто его одного мало для бюджета токенов
