# План: дубликаты ключей опций + защита facets-схемы

## Контекст находки

Запрос «найди черные розетки двухместные» дал 4 товара. По логам видно две независимые проблемы:

**1. Дубликаты ключей внутри одного логического свойства.**
В union-схеме после слияния «Розетки» + «Розетки силовые» одновременно живут:
- `cvet__tүs` (caption «Цвет», категория «Розетки»)
- `cvet__tүsі` (caption «Цвет», категория «Розетки силовые»)

Это **разные ключи в API** — фильтр `options[cvet__tүs][]=Чёрный` находит товары только из первой категории, на вторую не действует. FilterLLM выбирает один ключ, второй не используется → теряется бо́льшая часть ассортимента. Таких пар немного, но они есть (цвет — самый частый случай).

**2. Facets API в этом запросе вернул options с пустыми `values`.**
В логах:
```
[CategoryOptionsSchema] "Розетки": 63 keys, 0 values (from 1000 products, 2850ms)
[CategoryOptionsSchema] "Розетки силовые": 34 keys, 0 values (from 54 products, 354ms)
```
Текущая ветка fallback в `getCategoryOptionsSchema` срабатывает только если `optionsArr.length === 0`. Когда API отдаёт ключи без значений (или мы их теряем при парсинге), мы кладём в кеш битую схему на 30 минут и FilterLLM не может ничего валидировать — каждый ответ LLM отбрасывается как «value not in schema».

Лог `[FilterLLM] Schema (78 keys): cvet__tүs (cvet__tүs):` (после двоеточия пусто) — подтверждает: values пустые.

## Что меняем

### Этап 1 — починить пустые values в facets (блокирующий)

Файл: `supabase/functions/chat-consultant/index.ts`, функция `getCategoryOptionsSchema`.

1. Считать `totalValues` по ходу парсинга `options`.
2. Если **`totalValues === 0`** при непустом `optionsArr` — это битый ответ. Не кешировать, не возвращать, упасть в `getCategoryOptionsSchemaLegacy`.
3. Не кешировать также пустые/частичные результаты legacy (`schema.size === 0`) — сейчас кешируется любой результат, в т.ч. пустой, на 30 минут.
4. В `[CategoryOptionsSchema]` логе вывести honest счётчик `values=N` рядом с `keys=N` (сейчас в одной из веток считается, в другой — нет).

Доп. диагностика: в `schemaDebug` логе печатать первые 3 значения каждого ключа (сейчас печатает пусто, потому что values правда пусты — после фикса появятся).

### Этап 2 — склейка дубликатов ключей (главное изменение)

Дубликат — это пара ключей с **одинаковым нормализованным caption** (или нормализованным "корнем" ключа без хвоста-транслита казахского). Примеры:
- `cvet__tүs` ↔ `cvet__tүsі` → caption «Цвет» в обоих
- `kolichestvo_postov__saңylaular_sany` ↔ возможные варианты — caption «Количество постов»

Подход — без хардкода списка пар, через нормализацию.

**A. Построить группы синонимов на этапе union-схемы.**

В `getUnionCategoryOptionsSchema` (а также при работе с одиночной схемой в `getCategoryOptionsSchema`) после сборки `union: Map<key, {caption, values}>`:

1. Сгруппировать ключи по `normalizeCaption(caption)` — нижний регистр, убрать ё→е, пробелы, скобки, единицы измерения в скобках.
2. Внутри группы из 2+ ключей — это эквивалентные ключи. Объединить их `values` (union множеств), но **сохранить список оригинальных ключей** в новой структуре:
   ```
   { caption, values: Set, equivalentKeys: string[] }
   ```
3. Один из ключей выбираем «представителем» (используется как имя в схеме для LLM), остальные считаются alias-ами. Представитель — ключ с наибольшим числом значений (самый «полный»).
4. В FilterLLM-схему отдавать **только представителя**: LLM не должна видеть две одинаковых строки. Это сразу убирает её замешательство и экономит токены.

**B. На применении фильтра — отправлять во все эквивалентные ключи.**

В `executeSearch` (строка ~3314) и во всех аналогичных местах, где формируются `options[key][]`:

```ts
for (const [key, value] of Object.entries(resolvedFilters)) {
  const equivalents = equivalentKeysMap.get(key) || [key];
  for (const eqKey of equivalents) {
    params.append(`options[${eqKey}][]`, value);
  }
}
```

API 220volt при нескольких `options[K1][]=V & options[K2][]=V` объединяет результаты по ИЛИ внутри разных ключей характеристик (по документации) → ровно то поведение, что нужно: товар с заполненным любым из эквивалентных ключей попадёт в выдачу.

**C. Прокинуть карту эквивалентов от схемы к executeSearch.**

Сейчас `resolvedFilters: Record<string, string>` плоский. Чтобы не ломать сигнатуры, заведём отдельный side-channel: при сборке схемы класть карту в новый параметр `optionAliases: Map<representativeKey, string[]>` рядом с `resolvedFilters` и протаскивать в `executeSearch`. Без LLM-логики: чисто механическая мультипликация ключей при формировании HTTP-параметров.

### Этап 3 — гарантия для цвета (стабилизатор)

Дополнительно, как страховку и "мостик" пока не убедились, что нормализация caption ловит ВСЕ цветовые дубли:

В `equivalentKeysMap` после автогруппировки добавить статичный merge всех ключей, **которые начинаются с `cvet`** (но НЕ `cvetovaya_temperatura`/`cvetnost`/прочие). Это ловит даже редкий случай, когда caption по какой-то причине различается.

Логика: «всегда считаем все `cvet*`-ключи (кроме явно не-цветовых) одной семьёй».

### Этап 4 — диагностика и тесты

1. Лог при склейке: `[OptionAliases] grouped: cvet__tүs ↔ cvet__tүsі (caption="Цвет", repr=cvet__tүsі because 12 values vs 4)`.
2. Лог при применении: `[Search] Applying filter cvet=Чёрный via 2 alias keys: [cvet__tүsі, cvet__tүs]`.
3. Прогнать руками три запроса:
   - «черные розетки двухместные» — ожидаем больше 4 товаров.
   - «белые выключатели» — проверяем, что `cvet`-семья работает и в другой категории.
   - «теплый свет лампа e27» — НЕ должно склеиваться `cvet*` с `cvetovaya_temperatura` (контроль ложноположительных).

## Что НЕ трогаем

- Промпт FilterLLM (Точка 8 про цвет vs цветовую температуру) — оставляем как есть, только убираем из схемы дубль, чтобы LLM не путалась.
- Поведение `chat-consultant` outside того места, где формируется итоговый запрос к API.
- Endpoint `category_facets` в `search-products/index.ts` — он отдаёт сырые данные API без склейки, и это правильно (он публичный helper).

## Технические детали (кратко)

Файлы:
- `supabase/functions/chat-consultant/index.ts`:
  - `getCategoryOptionsSchema` — добавить детект `totalValues===0` и no-cache для битых ответов (~строки 1330-1360).
  - `getUnionCategoryOptionsSchema` — построить группы эквивалентов, выбрать представителей, вернуть `{ schema, optionAliases }` (~1433-1454).
  - Все места вызова union-схемы — принять и протащить `optionAliases`.
  - `executeSearch` (~3314) — мультипликация `options[]` через `optionAliases`.
- Новый helper `normalizeOptionCaption(caption: string): string` рядом с `cleanOptionCaption`.

Ничего в БД, в edge-config, в клиенте не меняется. Изменения локализованы внутри `chat-consultant` + ноль изменений API контракта `search-products`.

Риски:
- Ложная склейка двух разных смыслов с похожим caption. Mitigation: нормализация caption + ограничение «склеивать только если key-prefix до первого `__` совпадает». То есть `cvet__tүs` и `cvet__tүsі` оба начинаются с `cvet__` → склеиваем; `cvet__tүs` и `cvetovaya_temperatura__...` начинаются с `cvet*` но префикс до `__` разный → НЕ склеиваем. Это даёт защиту от точки 8 промпта на инфраструктурном уровне.
