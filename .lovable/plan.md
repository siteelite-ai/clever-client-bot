

# Итоговый план реализации — Главный архитектор

## Диагноз: две корневые причины нестабильности

Изучив код `chat-consultant/index.ts` (4337 строк), я подтверждаю выводы всех пяти аудиторов и выделяю **два конкретных дефекта**, которые порождают все наблюдаемые симптомы:

### Дефект 1: Загрязнение контекста в `generateSearchCandidates`

**Где:** строка 1610 — `slice(-10)` тащит 10 сообщений истории, строка 1622 — инструкция «АНАЛИЗИРУЙ С УЧЁТОМ КОНТЕКСТА».

**Почему ломает:** Классификатор (строка 580) уже имеет правильную изоляцию: «Если сообщение — САМОСТОЯТЕЛЬНЫЙ НОВЫЙ ЗАПРОС, извлекай ВСЕ поля ТОЛЬКО из текущего сообщения». Но `generateSearchCandidates` — это **отдельный** LLM-вызов со своим промптом, где такой изоляции нет. Поэтому после разговора про «прайс» новый товарный запрос маршрутизируется как продолжение темы «прайса».

**Следствие:** Бот говорит «общего прайса нет» вместо поиска товара.

### Дефект 2: Отсутствие доменного разделения в ранжировании

**Где:** `scoreProductMatch` (строка 1417) и `rerankProducts` (строка 1466).

**Почему ломает:** Функция считает score по токенам, спецификациям и бренду (0-100). Но слово «розетка» есть и в силовых, и в телефонных/компьютерных розетках. Нет штрафа за несоответствие домена. Если телефонная розетка случайно набирает больше token-баллов (например, «двойная» совпало) — она выходит в топ.

**Следствие:** Бот предлагает RJ11/RJ45 вместо электрических розеток.

### Дефект 3 (усилитель): Недетерминированный финальный ответ

**Где:** строка 4063-4067 — вызов LLM без `temperature`.

**Следствие:** Один и тот же набор товаров может быть представлен по-разному. Иногда бот «додумывает» и подменяет результаты.

## План реализации: 2 этапа

### Этап 1 — Высокий эффект, минимальный риск

**1.1. `temperature: 0` для финального ответа**
- Строка ~4063: добавить `temperature: 0` в объект запроса
- Риск: нулевой. Делает ответы стабильными

**1.2. Изоляция контекста в `generateSearchCandidates`**
- Строка ~1610-1622: если классификатор определил `product_category` — обнулить историю в промпте candidates
- Конкретно: передавать `classification` как параметр функции, и если `classification.product_category` заполнена — не включать `historyContext` в промпт
- Уточняющие диалоги (строки 1624-1636) продолжат работать: они срабатывают когда `product_category` НЕ заполнена (короткие ответы типа «черную», «IP44»)
- Риск: низкий. Классификатор уже проверен и работает корректно

### Этап 2 — Доменный guard и ранжирование

**2.1. Мягкий доменный штраф в `rerankProducts`**
- В `scoreProductMatch` (строка 1417): если пользователь не упоминал telecom-ключи (`RJ11`, `RJ45`, `телефон`, `интернет`, `компьютер`, `LAN`, `data`), а в названии товара они есть — штраф -30 баллов
- И наоборот: если пользователь явно ищет telecom — штрафовать обычные силовые розетки
- Реализация: новая функция `domainPenalty(product, userQuery)`, вызываемая из `scoreProductMatch`
- Риск: средний. Нужно аккуратно составить списки ключевых слов

**2.2. Активация Quick Filters для «постов/мест»**
- Использовать `extractQuickFilters` для детерминированной фильтрации количества мест: «двухместная» → filter по `2`, «трёхместная» → `3`
- Если Quick Filter не совпал — товар исключается из выдачи
- Риск: средний. Нужно убедиться что ключи в API-данных корректно маппятся

**2.3. Мягкий fallback вместо пустого ответа**
- Если после domain guard и Quick Filters осталось 0 товаров — показать лучшие результаты без фильтров с пометкой: «Точного совпадения не найдено. Вот ближайшие варианты:»
- Добавить инструкцию в системный промпт: запрет подменять электрическую розетку телефонной

## Регресс-тесты после каждого этапа

1. Чистая сессия: «нужна черная двухместная розетка» → электрические розетки
2. Загрязнённый контекст: 3 сообщения про «прайс» → тот же запрос → товарный поиск
3. Повтор в сессии: запрос после 2 других ответов → стабильный результат
4. Telecom: «телефонная розетка RJ11» → telecom-товары не режутся
5. Длинный диалог: 7+ сообщений → новый запрос работает чисто
6. Уточнение: «розетки» → бот уточняет → «черную» → корректная фильтрация
