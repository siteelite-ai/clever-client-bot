

# Консилиум архитектора по плану «totalMatched vs displayed»

Прогнал план через три роли: Backend Architect, Search/Retrieval Engineer, Code Reviewer. Вердикт: **план безопасный, ничего не ломаем, но есть 2 нюанса, которые нужно зафиксировать до имплементации**.

## TL;DR

План минимально-инвазивный: мы НЕ меняем поиск, фильтры, suppress-query, роутер, лимиты API. Меняем только **что показываем в промпт LLM**: вместо «длина обрезанного списка» → «реальное число до обрезки». Риск задеть другие функции — низкий, потому что `displayed` остаётся ровно тем же массивом, что и сейчас.

## Что в плане однозначно безопасно

1. **`displayed` = `foundProducts.slice(0, 15)`** — точно тот же массив, что улетает в LLM-контекст сейчас. Карточки, ссылки, цены, порядок — без изменений.
2. **`pickDisplayWithTotal` — чистая функция**, без сайд-эффектов. Заменяет `.slice(0, 15)` 1:1.
3. **Цена-intent ветка уже работает по этой схеме** (`priceIntentClarify.total`). Мы просто распространяем её паттерн на остальные ветки — это и есть «единый контракт», к которому вели прошлые консилиумы.
4. **Лимит API (`per_page=30`, мердж нескольких категорий) не трогаем** — `totalMatched` берётся из уже собранного `serverFiltered` / `merged`, никаких новых запросов в API.

## Риски, которые нужно закрыть в плане

### Риск 1 — `totalMatched` ≠ «всего в каталоге»

`serverFiltered.length` — это не «всего товаров в каталоге», а «сколько мы успели собрать в пределах `MAX_PAGES × PER_PAGE` по всем категориям ветки». Если реальная выдача в каталоге 500, а мы собрали 153 из-за пейджинга — бот скажет «нашлось 153», и это будет враньё в другую сторону.

**Митигация в плане:**
- Называть переменную честно: `totalCollected`, а не `totalMatched`.
- В промпте формулировать так: «Подобрано {totalCollected} вариантов» (а не «всего в каталоге»).
- Если `totalCollected === MAX_PAGES × PER_PAGE × categoriesCount` (упёрлись в потолок) — добавлять в промпт хинт «возможно, есть ещё, уточните запрос».

### Риск 2 — `fewProducts` порог завязан на displayed.length

Сейчас `fewProducts = productCount <= 7` срабатывает на длине обрезанного массива. Если переключить на `totalCollected <= 7`, то ветка «показать все» вообще перестанет триггериться при `totalCollected = 153`, что и нужно. Но есть пограничный кейс: `totalCollected = 8`, `displayed = 8` — раньше шла ветка «много», теперь пойдёт ветка «мало» (это правильно, но проверить регрессом).

**Митигация в плане:**
- `fewProducts = totalCollected <= 7` — оставляем как в плане.
- Регресс-кейс: запрос ровно с 8 товарами, убедиться что показаны все 8 без фразы «всего нашлось».

## Чего в плане не хватает

1. **Переименовать `totalMatched` → `totalCollected`** — честнее отражает суть (мы посчитали то, что собрали, а не весь каталог).
2. **Хинт «упёрлись в потолок пейджинга»** — если `totalCollected` равно теоретическому максимуму ветки, добавить в промпт «возможно есть ещё».
3. **Регресс-кейс на границе 7/8 товаров** — убедиться что `fewProducts` ветка работает корректно.

## Что НЕ трогаем (повторно подтверждаем)

- `suppressResolvedFromQuery` и его 7 точек вызова — последний фикс, без изменений.
- Роутер веток (slot / category-first / matcher / bucket-N / replacement) — без изменений.
- API-лимиты (`MAX_PAGES`, `PER_PAGE`) — без изменений.
- Markdown-формат карточек, top-3 в финальном ответе — без изменений.

## Итоговый вердикт

План **одобряем к реализации с двумя поправками**:

1. Переменная называется `totalCollected`, формулировка в промпте — «Подобрано {N} вариантов».
2. Добавить хинт «возможно есть ещё» при упоре в потолок пейджинга.

Риск задеть suppress-query, роутер, slot-ветку, replacement — **нулевой**, потому что `displayed` идентичен текущему `foundProducts`. Меняется только текст в промпте + один новый числовой параметр.

## Что меняем в файле

`supabase/functions/chat-consultant/index.ts`:
- константа `DISPLAY_LIMIT = 15` в шапке
- helper `pickDisplayWithTotal(all, limit)` рядом с `suppressResolvedFromQuery`
- 12 точек замены `.slice(0, 15)` → `pickDisplayWithTotal(...)`
- в каждой ветке сохранить `totalCollected` в общий контекст
- в промпте (строки ~5214–5243) использовать `totalCollected` для «Подобрано N», `displayed.length` для «покажи top-3»
- лог `[Chat] DisplayLimit: collected=… displayed=… branch=… cappedByPaging=true|false`

Регресс после деплоя:
1. «Розетки INSPIRIA» → «Подобрано ~150, показываю первые 3, уточните цвет/серию»
2. Узкий запрос с 2–3 товарами → показаны все, фразы «подобрано» нет
3. Запрос ровно с 8 товарами → ветка `fewProducts`, показаны все 8
