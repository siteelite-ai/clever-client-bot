

# Slot-память сломана: уточняющий ответ классифицируется как «новый запрос», и контекст первого хода теряется

## Что я нашёл в логах

**Ход 1**: «нужна черная двухместная розетка» → бот задал уточняющие вопросы (заземление? шторки?).
**Ход 2**: «Заземление не важно, защитные шторки тоже» → бот ответил «нет готовой двухместной чёрной» — **галлюцинация**, потому что в каталоге такие розетки есть.

В логах ровно один корневой строчка:

```
[AI Candidates] Context ISOLATED: new product query detected (category="розетка"), history pruned
```

И сразу после:
```
[AI Candidates] Extracting search intent from: "Заземление не нужно", classificationCategory: розетка
```

intent-extractor получил **только** фразу «Заземление не нужно» БЕЗ слов «двухместная, чёрная, скрытая». Поэтому сгенерировал пустых кандидатов `[розетка, вилка, удлинитель, колодка, сетевой фильтр]`. Поиск нашёл «колодки 2 гнезда чёрные **открытой установки**» (переносные, не скрытые) — это и был не тот товар. LLM честно ответил «нет готовой».

## Корень в коде

`supabase/functions/chat-consultant/index.ts`, строки 1600-1614:

```ts
const isNewProductQuery = !!classificationCategory;
const recentHistory = isNewProductQuery ? [] : conversationHistory.slice(-10);
```

Логика: «если классификатор определил категорию — выкидываем историю». Но классификатор **всегда** определяет категорию для уточняющих ответов в товарном диалоге (потому что и в первом, и во втором ходу обсуждаются розетки). Условие `!!classificationCategory` ловит и **настоящий новый запрос**, и **уточнение в рамках старого** — между ними не различается.

## Что это значит технически

У нас сейчас детектор «нового запроса» работает по принципу «есть категория → новый», что эквивалентно **отключению истории для всех товарных уточнений вообще**. Это ломает любой 2-ходовой диалог про подбор товара: ход 1 — описание, ход 2 — ответ на уточняющий вопрос → контекст потерян.

В коде (строки 1620-1632) уже есть промт-блок «ОБРАБОТКА УТОЧНЯЮЩИХ ОТВЕТОВ», который специально учит LLM восстанавливать контекст. Но он **никогда не срабатывает**, потому что ему просто не подают history — её обрезали раньше на уровне условия.

## Фикс (точечный, ~30 строк)

### Шаг 1. Различать «новый запрос» и «уточнение в рамках старого»

Заменить `isNewProductQuery = !!classificationCategory` на двухфакторную проверку:

```ts
// Это уточнение, если:
// (a) последняя реплика ассистента содержала уточняющий вопрос (?, "уточните", "нужно ли")
// (b) И категория текущего запроса совпадает с категорией предыдущего товарного хода
const lastAssistantMsg = [...conversationHistory].reverse().find(m => m.role === 'assistant')?.content || '';
const looksLikeClarificationFollowup = 
  /\?|уточни|нужно ли|какой|какая|какие|для каких|с\s+каким/i.test(lastAssistantMsg.slice(-500));

const previousCategory = extractCategoryFromHistory(conversationHistory); // последняя category из логов истории
const sameCategory = previousCategory && classificationCategory && 
  previousCategory.toLowerCase() === classificationCategory.toLowerCase();

const isFollowup = looksLikeClarificationFollowup && sameCategory;
const isNewProductQuery = !!classificationCategory && !isFollowup;
```

Если `isFollowup = true` — **history НЕ обрезается**, intent-extractor получает контекст и сам мёрджит «двухместная чёрная» из первого хода + «без заземления, без шторок» из второго. Этот сценарий уже описан в промте на строках 1620-1632 — он сразу заработает.

### Шаг 2. Добавить slot-память в логи

Логировать что попало в intent-extractor:

```
[AI Candidates] Followup detected: lastAssistantQ=true, sameCategory=true → history KEPT (8 msgs)
[AI Candidates] Merged context: prev_query="черная двухместная розетка" + current="без заземления"
```

### Шаг 3. Safety net — если `isFollowup`, intent-extractor ОБЯЗАН вернуть intent="catalog"

В случае followup'а если LLM по ошибке вернул `intent="general"` — форсировать `catalog` (уже описано в промте, но добавим хард-фолбэк в коде).

## Что НЕ делаем

- Не трогаем sampling, модели, max_tokens, RAG, persona, widget — они сейчас правильные
- Не меняем промт intent-extractor'а — там уже есть правильные инструкции про followup
- Не вводим полноценную slot-state машину с БД — overkill, текущий промт + history уже всё умеет, надо просто перестать обрезать ему руки
- Не убираем ISOLATED полностью — он нужен когда пользователь после розеток внезапно спрашивает «а покажи светильники» (тут sameCategory=false → ISOLATED сработает корректно)
