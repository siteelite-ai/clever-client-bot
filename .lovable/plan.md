

# План: Серверная фильтрация через API options вместо локальной

## Суть идеи

Ты абсолютно права. API 220volt уже поддерживает фильтрацию по характеристикам на стороне сервера: `options[cvet__tүsі][]=чёрный`. Функция `searchProductsByCandidate` уже принимает `resolvedFilters` и добавляет их как query-параметры. Но category-first ветка это **не использует** — она получает 50 товаров без фильтров и пытается отфильтровать локально.

## Почему текущий подход ломается

```text
category="розетки" → 2336 товаров → API отдаёт первые 50
→ среди этих 50 нет чёрных двухместных
→ локальный фильтр: 0 совпадений
→ бот думает, что таких товаров нет (а они есть, просто не попали в первые 50)
```

## Новый подход: каскадная серверная фильтрация

```text
1. resolveFiltersWithLLM — получаем ключи: {cvet=чёрный, kol_razemov=2}
2. API: category="розетки" + options[cvet][]=чёрный + options[kol_razemov][]=2
   → сервер ищет по ВСЕМ 2336, возвращает точные совпадения
3. Если 0 → ослабляем: убираем один фильтр, пробуем снова
```

## Проблема: курица и яйцо

resolveFiltersWithLLM нуждается в списке товаров, чтобы построить схему опций (какие ключи существуют, какие значения). Без товаров он не знает, что ключ цвета называется `cvet__tүsі`.

## Решение: двухэтапный подход

```text
Этап 1: category="розетки", per_page=50, без фильтров
  → получаем 50 товаров ТОЛЬКО для построения схемы опций
  → resolveFiltersWithLLM(эти 50, модификаторы) → {cvet__tүsі: "чёрный", kol_razemov: "2"}

Этап 2: category="розетки" + options[cvet__tүsі][]=чёрный + options[kol_razemov][]=2
  → API фильтрует по ВСЕМ 2336 товарам серверно
  → возвращает точные совпадения

Fallback: если этап 2 дал 0
  → убираем фильтры по одному и повторяем API-запрос
```

## Что меняется в коде

**Файл**: `supabase/functions/chat-consultant/index.ts`, category-first ветка (строки 3219-3350)

### Изменение 1: Первый запрос — только для схемы
Текущий запрос `searchProductsByCandidate({category: plural}, token, 50)` остаётся как есть, но его результат используется **только** для resolveFiltersWithLLM (построение схемы опций).

### Изменение 2: Второй запрос — с серверными фильтрами
После resolveFiltersWithLLM делаем **новый** API-запрос с `resolvedFilters`:
```typescript
const serverFiltered = await searchProductsByCandidate(
  { query: null, category: pluralCategory, ... },
  token, 50,
  resolvedFilters  // ← передаём как options[key][]=value
);
```

### Изменение 3: Каскадный fallback
Если с полным набором фильтров 0 — убираем фильтры по одному и повторяем API-запрос (не локальный фильтр, а именно серверный).

### Изменение 4: Убрать локальный AND-фильтр
Локальный `applyAndFilter` больше не нужен как основной механизм — сервер делает эту работу. Можно оставить как финальную проверку, но не как основной фильтр.

## Что НЕ трогаем
- resolveFiltersWithLLM — работает, строит схему и резолвит модификаторы
- Classify — работает
- toPluralCategory — работает
- Другие ветки (title-first, article-first, price) — без изменений

## Ожидаемый результат
Запрос "черная двухместная розетка":
1. Schema: 50 розеток → схема опций → `{cvet: "чёрный", razem: "2"}`
2. API: `category=розетки&options[cvet][]=чёрный&options[razem][]=2` → ищет по всем 2336
3. Результат: чёрные двухместные розетки BRITE
