

# План: Адаптация к улучшенному API 220volt.kz

## Что изменилось в API

Разработчик внес два улучшения:

1. **Параметр `query`** — теперь ищет по названию, длинному названию, описанию и контенту товара (раньше индексация была неполной)
2. **Фильтр `options[][value]`** — документирован универсальный формат фильтрации по любой опции товара: `options[key][]=value`

## Что нужно сделать

### 1. Протестировать улучшенный поиск

Прогнать через API тот же проблемный запрос (`query="ECO T8 6500К"`) и убедиться, что лампа теперь находится. Если находится — текущая логика уже будет работать корректно без изменений кода.

### 2. Улучшить промпт LLM 1 — включать модельные коды в query

Независимо от результатов теста, нужно улучшить `extractionPrompt` в `chat-consultant/index.ts`:
- Добавить инструкцию: модельные коды (T8, A60, MR16, G45) ОБЯЗАТЕЛЬНО включать в `query`-кандидат
- Добавить пример для ламп с техническими параметрами в секцию "СИНОНИМЫ ТОВАРОВ"

Это повысит качество поиска, раз `query` теперь действительно ищет по полному названию.

### 3. Обновить стратегию: query с характеристиками

Сейчас в промпте правило: "НЕ включай характеристики в query!". Раз API теперь поддерживает поиск по содержимому, имеет смысл **ослабить** это ограничение — разрешить включать ключевые характеристики (мощность, температуру) в `query` наряду с `option_filters`.

### 4. Добавить Sibling Article Construction (fallback)

Если после Pass 2 = 0 результатов, но в Pass 1 есть "родственные" товары (тот же артикул с другой характеристикой), попытаться сконструировать артикул и поискать по нему. Это страховка на случай, если `query` всё ещё не покрывает 100% товаров.

## Файлы для изменения

- `supabase/functions/chat-consultant/index.ts` — промпт LLM 1 + fallback логика
