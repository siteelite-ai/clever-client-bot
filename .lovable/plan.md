## Архитектурный разбор: что не так на самом деле

Сначала жёстко отделю «симптомы» от «корней». Симптом — LLM выбрала `Вид розетки` вместо `Количество разъемов`. Корень — другое.

### Что есть сейчас (как реально работает один проход FilterLLM)

```text
                       ┌─────────────────────────────────────────┐
   user modifiers ───► │  ОДИН вызов LLM с ОДНИМ системным       │
   ["чёрный",          │  промптом, который содержит:            │
    "двухгнездовая"]   │   - ВСЕ 53 ключа категории              │
                       │   - ВСЕ значения каждого ключа          │
                       │   - общую инструкцию "сопоставь"        │
                       │  → ответ {"filters": {"key": "value"}}  │
                       └─────────────────────────────────────────┘
                                       │
                                       ▼
                       Жёсткая валидация: ключ ∈ schema, значение ∈ values
                       → если нет — UNRESOLVED, дальше как text query
```

### Системные дефекты этой архитектуры

1. **«One-shot resolver» против «множественной семантики».** Мы просим одну LLM за один вызов одновременно: (а) распознать физическое свойство модификатора, (б) выбрать один ключ из 53 кандидатов, (в) выбрать одно значение из 5–60. Если ошиблась в (а) — всё дальше мусор. Никакой «второй попытки на других кандидатах» нет.
2. **Нет понятия «семантического пространства характеристики».** Мы передаём ключи как плоский список с captions. У модели нет промежуточного представления вида «эта характеристика — про КОЛИЧЕСТВО / ЦВЕТ / РАЗМЕР / ТИП-МОНТАЖА / БРЕНД». Поэтому она каждый раз заново угадывает по caption-у. Это и есть «хардпромптинг по словам».
3. **Schema приходит обрезанной (legacy fallback на 30 товаров) и мы это маскируем.** Реальный facets API «Розетки» — 16 секунд. Наш timeout 6/8 секунд. Значит, в проде легаси-выборка дала 5 цветов вместо 61 и, возможно, без ключа `Количество разъемов` вообще. LLM делает «лучший выбор из мусора» — и мы её обвиняем.
4. **Резолвер не знает, что в категории МОЖЕТ быть несколько ключей одного класса.** В «Розетках» есть `Количество разъемов`, `Количество USB-портов`, `Количество в групповой упаковке` — все «количественные». Какой именно «двухгнездовая»? Сейчас это решает текстовая близость caption-у, что и приводит к промаху.
5. **Схема — это «словарь магазина», а не онтология свойств товара.** Мы каждый раз учим LLM с нуля, что `cvet__tүs` — это про цвет. Между запросами нет ничего, что переиспользовалось бы.

Итого: проблема не в timeout, не в промпте, не в синонимах. Проблема — в **отсутствии слоя онтологии характеристик** и в **отсутствии маршрутизации модификаторов по этой онтологии**.

## Системное решение: Property Graph + двухступенчатый резолвер

Вместо «один LLM-вызов на всё» — три явных слоя, каждый со своей зоной ответственности и наблюдаемостью.

### Слой 1. Schema Provider (источник правды о категории)

Один интерфейс, который для категории отдаёт **полную, не-обрезанную** схему опций. Реализация скрывает: время ответа API, retry, кеш, prefetch, fallback на legacy. Контракт наружу — `Promise<CategorySchema>` + поле `confidence: 'full' | 'partial' | 'empty'`.

Ключевая идея — **никакой код выше не должен знать, пришла ли схема из facets-API или из legacy-сэмпла**. Если `confidence !== 'full'`, дальнейшие слои ведут себя по-другому (см. слой 3 — отказ от резолвинга вместо угадывания на мусоре).

Это убирает «маскировку деградации» из текущего кода: сейчас legacy возвращает 5 значений и притворяется полной схемой.

### Слой 2. Property Classifier (онтология поверх API-ключей)

Новый слой, которого сейчас нет. Превращает «голый» schema-ключ в **типизированное свойство товара**:

```text
        raw API key                  →   semantic property
   ────────────────────────────         ──────────────────────────
   cvet__tүs                       →   { class: "color",
                                         valueDomain: "categorical",
                                         exemplars: [чёрный, белый…] }
   kolichestvo_razyemov_*          →   { class: "count",
                                         subject: "разъём/гнездо/гнезд",
                                         valueDomain: "integer",
                                         exemplars: [1..8] }
   kolichestvo_usb_portov_*        →   { class: "count",
                                         subject: "USB-порт",
                                         valueDomain: "integer" }
   vid_rozetki_*                   →   { class: "type",
                                         valueDomain: "categorical-named",
                                         exemplars: [электрическая, USB, TV…] }
   tip_montagha_*                  →   { class: "mounting" }
   napryaghenie_*                  →   { class: "physical-quantity",
                                         unit: "В" }
   stepeny_zaschity_*              →   { class: "ip-rating" }
   brend_*                         →   { class: "brand" }
```

Эта классификация делается **один раз на категорию** при загрузке схемы (LLM-вызов по 53 ключам, кешируется на сутки). Это и есть онтология. Дальше она переиспользуется на каждом пользовательском запросе — без LLM, без гадания.

Важно: классификатор смотрит на (caption, sample-of-values, key-tokens) и возвращает структурированный класс из ЗАКРЫТОГО списка (10–15 классов: `color | count | type | mounting | ip-rating | brand | series | physical-quantity | boolean-feature | material | dimension | …`). Закрытый список — потому что онтология предметной области (электротовары) конечна.

### Слой 3. Modifier Router → Resolver (двухступенчатый)

Вместо «дай LLM 53 ключа и угадай» — два явных шага:

**Шаг A. Routing (cheap).** Маленький LLM-вызов: «дан модификатор + закрытый список semantic-классов категории — определи класс модификатора». На вход — 10–15 классов, не 53 ключа. Ошибиться сложно.

```text
   "двухгнездовая"   →   { class: "count", subject_hint: "разъём/гнездо" }
   "чёрный"          →   { class: "color" }
   "16А"             →   { class: "physical-quantity", unit_hint: "А" }
   "IP54"            →   { class: "ip-rating" }
```

**Шаг B. Value matching внутри класса (детерминированно, без LLM).** Когда класс выбран, мы знаем подмножество ключей с тем же классом. Если их 1 — берём его, идём в значения. Если их несколько (например, два count-ключа в розетках) — арбитраж по `subject_hint` через простой scoring (caption similarity), никаких догадок про вид/тип.

```text
   class=count, subject_hint="разъём/гнездо"
       candidates: [kolichestvo_razyemov ("разъемов"),
                    kolichestvo_usb_portov ("USB-портов"),
                    kolichestvo_v_gruppovoy ("в упаковке")]
       score by token overlap with subject_hint:
         "разъемов" ↔ "разъём/гнездо" → 1.0
         "USB-портов"                  → 0.0
         "в упаковке"                  → 0.0
       → kolichestvo_razyemov
   value mapping: "двух" → numeral table → 2 → ∈ {1..8} → ✓
```

Числительное-в-цифру и цвет-в-цвет — это уже есть в коде (numeralMap, стем-матчинг). Эти куски НЕ выбрасываем, они становятся value-matchers внутри своего класса.

**Шаг C. Только если шаг B не справился (амбигуация).** Один узкий LLM-вызов: «модификатор X не лёг на единственный ключ; вот 2–3 кандидата с реальными значениями — какой?». Это резервный путь, не основной.

### Принципиальное отличие от прежнего плана

Прежний (плохой) план: добавить в один большой промпт ещё одно правило-словарь («гнездо ≈ разъём»). Это хардкод предметной области в промпте. Завтра придёт «трёхполюсный автомат» — допишем ещё. Послезавтра «E27» — ещё. Промпт превращается в свалку.

Новый: **онтология свойств — отдельный слой, считается один раз на категорию из реальных данных API**. LLM-резолвер становится тонким: «класс → ключ → значение». Никакого знания предметной области в промптах не зашивается.

## Деградация и наблюдаемость (обязательная часть)

Нынешний скрытый дефект — мы не видим, насколько хороша полученная схема. Это исправляется на уровне Schema Provider:

- Каждая схема несёт `confidence: full | partial | empty`.
- `partial` (legacy-выборка) → Resolver НЕ запускается; пользователю явно сообщается «каталог сейчас отвечает медленно, покажу топ результатов, уточните вручную». Сейчас мы вместо этого тихо отдаём LLM мусор и принимаем её ошибку как нашу.
- `full` → нормальный путь.
- `empty` → дегрейд по уже существующей DEGRADED_UX-ветке.

Логи унифицируются: `[Resolver] cat=X schema_confidence=full classes=[color,count,brand,…] mod="двухгнездовая" → class=count → key=kolichestvo_razyemov → value=2`. Один человекочитаемый трейс на модификатор.

## Что это даёт в долгую

- **Масштабируемо.** Появилась новая категория с 80 опциями — ничего в коде не трогаем. Property Classifier один раз классифицирует, дальше работает.
- **Не хардкодит словари.** Никаких «гнездо = разъём» в промптах.
- **Чинит проблему таймаута на уровне правды, а не симптома.** Если schema partial — мы это знаем и не врём пользователю; если full — резолвинг работает по полной онтологии.
- **Сохраняет всё текущее.** numeralMap, стем-матчинг, suppressResolvedFromQuery, DEGRADED_UX, FilterLLM-bulkhead, кеш категорий — всё переиспользуется, просто переезжает в свои слои.

## Технический раздел (что физически меняется в коде)

Файл `supabase/functions/chat-consultant/index.ts`, без новых файлов.

1. **Schema Provider.** Текущая `getCategoryOptionsSchema` оборачивается типом `CategorySchema { schema, confidence, source, productCount }`. `confidence='full'` только если пришли facets-API + values>0. Legacy-результат — всегда `'partial'`. Таймаут facets-API поднимается под реальные ~16s, но это вторично: главное — флаг confidence.

2. **Property Classifier.** Новая функция `classifyCategoryProperties(schema): Map<apiKey, PropertyClass>`. Внутри — один LLM-вызов на всю категорию (закрытый список классов в системе), результат кешируется на сутки рядом с `categoryOptionsCache`. Класс — узкий enum.

3. **Modifier Router.** Новая функция `routeModifiers(modifiers, propertyClasses): Array<{modifier, class, subjectHint?}>`. Один дешёвый LLM-вызов: модификаторы → классы (только классы, не ключи).

4. **Value Resolver.** Новая функция `resolveByClass(routed, schema, propertyClasses)`. Детерминированная: фильтрует ключи по классу, скорит по subjectHint, мапит значение через существующие numeralMap/стем-матчер/прямое сравнение.

5. **Ambiguity Fallback.** Если на шаге 4 в классе >1 ключа и scoring неуверен (близкие баллы) — точечный LLM-вызов «какой из этих 2 ключей?». Это и есть «второй проход», но узкий.

6. **`resolveFiltersWithLLM`** становится оркестратором этих 4 шагов; старая «one-shot простыня» удаляется. Существующая обратная совместимость API (`{resolved, unresolved}`) сохраняется — выше по стеку ничего не меняется.

7. **DEGRADED_UX-ветка** срабатывает теперь не только по «zero values», но и по `schema.confidence='partial'` — потому что резолвить по обрезанной схеме = врать.

После твоего апрува переключаюсь в build-режим и реализую слой за слоем, начиная со Schema Provider + флага confidence (это уже починит большую часть прода), затем Property Classifier, затем Router/Resolver. Каждый слой — отдельный коммит, каждый — самостоятельно тестируемый.
