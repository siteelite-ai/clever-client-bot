
# Консилиум по чистоте спецификации

Состав: **Архитектор**, **Prompt-инженер**, **AI/ML-инженер**, **QA-инженер**.
Контекст: пользователь требует исключить из спецификации все локальные заплатки и точечные хардкоды («F4: убрать `kolichestvo_gnezd`», whitelist рабочих facet-ключей, seed-словари, числовые таблицы эквивалентов). Система должна работать **при любых условиях**, опираясь только на:
1. Реальный Catalog API 220volt и его документацию;
2. Грамотно написанные промпты, в которых LLM сама принимает решения по характеристикам, морфологии и эквивалентам.

---

## Вердикт консилиума по каждому подозрительному месту

### A. §9C.2 + ADR 28.19 — `facet_filter_whitelist_json` — **ХАРДКОД, удалить полностью**

**Архитектор:** «9 ключей, зашитых на основании одного дня аудита, + probe-cron «1 ключ в сутки» — это имитация автоматизации. Через месяц таблица протухнет, через год придётся переписывать вручную. Системный путь — не вести параллельный реестр того, что API сам про себя знает.»

**AI/ML-инженер:** «Решение `работает / не работает фильтр` принимается **по результату** одного-единственного факта: `total === 0` при заведомо валидном значении из схемы. Это не нужно ни вайтлистить, ни пробить заранее — это видно прямо в ответе API на тот же запрос, который мы и так делаем.»

**Prompt-инженер:** «А `query_fallback` с инжектом значения в `?query=` — это нарушение нашего же C-5 («никакой query-pollution»), которое мы пометили «единственным санкционированным исключением». Так не работает: либо инвариант, либо нет.»

### B. §9.3 — числовые эквиваленты (`двух=2=двойная=две=2-х=II`) — **полу-хардкод, переписать на принцип**

**Prompt-инженер:** «В промпте мы перечисляем конкретные пары. Это словарь, переодетый в инструкцию. Системно — формулировать **принцип** для LLM: "трейты, описывающие количество/полюсность/число элементов, нормализуй к числовому представлению, ориентируясь на тип значений в `schema.values[]` соответствующего facet". LLM Gemini это умеет — мы просто должны явно сказать «нормализуй число», а не диктовать таблицу.»

**QA:** «Это снимет жалобы вида "а если пользователь скажет «спаренная» — её же нет в нашем списке?".»

### C. §9.2b — seed-словарь lexicon на 50–100 записей — **легитимно, но с оговоркой**

**Архитектор:** «Lexicon — не патч, а доменное знание (бытовое название «кукуруза» физически не выводится из API; такого слова в каталоге нет нигде). Это категориальный бизнес-онтологический слой, без него любая LLM так же провалится. Но **seed «50–100 записей при первом деплое»** — это лишнее обещание. Системнее: стартуем с **0 записей**, словарь пополняется только через ревью реальных `unresolved_traits` из логов.»

**AI/ML-инженер:** «Согласен. Это убирает соблазн «давайте докинем известных слов сразу» и заставляет систему учиться на реальных данных.»

**Prompt-инженер:** «И при пустом словаре пайплайн обязан работать — проверим инвариантом и тестом.»

### D. F-4 в `.lovable/plan.md` («ключ `kolichestvo_gnezd` не существует — заменить на `kolichestvo_razyemov…`») — **НЕ хардкод**

**QA:** «Это правка тестов под реальные ключи API, а не патч в проде. Тесты должны соответствовать действительности. Оставляем как есть, переформулируем без слова «убрать», чтобы не путало.»

### E. §9C.1 (локальная сортировка) и §9C.3 (`price=0`) — **обходы, но честные**

**Архитектор:** «Эти обходы документируют реальные дефекты API и помечены ADR 28.17 как временные с probe-чекером на возврат фикса. Это не хардкод, а зафиксированная инженерная компенсация недостатка внешней системы. Оставляем.»

## Что меняется в спецификации (системно)

### 1. §9C.2 — переписать «Whitelist» → «Try-and-degrade по факту»

Убрать `facet_filter_whitelist_json`, probe-cron, query_fallback. Заменить на:

> **Try-and-degrade.** Catalog Search всегда отправляет резолвленные фильтры через `options[<key>][]=`, как описывает API-контракт. Если суммарный ответ имеет `total === 0` **и** в запросе было ≥1 `options[]`-фильтра — выполняется ровно один **degrade-step**: повтор запроса без последнего по приоритету `options[]`-фильтра. Приоритет фильтров — порядок их добавления (последний в `slot.applied_filters` уходит первым). Шаги повторяются максимум 2 раза. Каждый сброшенный фильтр попадает в `slot.degraded_filters[]` и выводится Composer'ом отдельной строкой: «Не удалось точно отфильтровать по «{caption}: {value}» — показал результаты без этого условия». Метрики: `facet_degrade_step_count` (histogram 0..2), `facet_degraded{key}` (counter).
>
> **Никаких whitelist'ов, probe-cron'ов и query-инъекций нет.** Решение принимается по факту ответа API, а не по предположению о ключе. Если 220volt починят `cvet__tūs` — система сразу начнёт фильтровать корректно, без релиза.

Удаляются:
- блок «Whitelist рабочих ключей» (1127–1131);
- `app_settings.facet_filter_whitelist_json` из §24 (строка 2128);
- метрики `facet_query_fallback_used`, `facet_filter_blocked_total`, `whitelist_probe_added` из §22.2;
- ADR 28.19 (заменяется на «отвергнут — см. §9C.2 Try-and-degrade»);
- инвариант C-5-исключение возвращается к строгой формулировке без «единственного исключения».

### 2. §9.3 — переформулировать числовые эквиваленты как принцип

Строка 721 сейчас:
> числовые↔словесные эквиваленты: `двух=2=двойная=две=2-х=II`, `трёх=3=тройная`…

Заменить на:

> **Числовая нормализация.** Если facet принимает числовые значения (видно по `schema.values[]`: `["1","2","3","4"]` и т.п.), а трейт пользователя выражает количество словесно («двойная», «спаренная», «на три места», «II», «трёх-», «двух-») — нормализовать к числу из `values[]`. Решение принимает LLM на основании типа значений facet'а; жёсткой таблицы соответствий не существует.

То же для морфологии RU/KK и `ё↔е` — оставить как принципы без перечислений.

### 3. §9.2b — Lexicon Resolver: пустой старт + чёткий инвариант

В блоке «Bootstrap словаря (MVP)» (строка 651):
- убрать «Seed (вручную) ~50–100 записей»;
- сформулировать: «Стартовое состояние `lexicon_json.entries = []`. Все записи попадают в словарь через ревью реальных `unresolved_traits`/`empty_results` из `chat_traces` (cron + admin approval, как описано). До первого ревью система обязана работать корректно при пустом словаре — закрепить инвариантом L5 и golden-тестом.»
- ADR 28.15 пересмотреть: вариант (a) → «**стартуем с 0**, накапливаем по логам». Это ленивее (b)/(c), что и просит пользователь.

Новый инвариант:
> **L5.** При `lexicon.entries = []` Lexicon Resolver возвращает `{expanded_traits: user_traits, canonical_tokens: [], applied_aliases: []}` без ошибок; пайплайн продолжается на исходных трейтах.

### 4. §25 Golden Tests — переписать под новые системные правила

- **TC-69, TC-71, TC-72, TC-84:** убрать упоминания `query_fallback`. Сценарий: фильтр с проблемным ключом (`cvet__tūs="чёрный"`) уходит в API как `options[]`, API возвращает `total=0`, Catalog Search делает 1 degrade-step (сбрасывает цвет), Composer выводит строку «Не удалось точно отфильтровать по «Цвет: чёрный»…». Метрика `facet_degrade_step_count=1`.
- **TC-76:** «капсульная лампочка g4» — то же поведение через degrade на `forma_kolby`.
- **TC-83 (новый/переименованный):** «degrade-step не запускается, если в запросе нет `options[]`-фильтров» (защита от лишних запросов).
- **TC-новый (L5):** свежая сессия с `lexicon_json.entries = []` → запрос «лампа кукуруза» отдаёт корректную деградацию (Lexicon ничего не делает, Facet Matcher не находит «кукуруза», `unresolved_traits` → уточнение или открытое перечисление).

## Принципы, которые мы фиксируем по итогу консилиума

1. **API — единственный источник правды о facet-ключах.** Никаких параллельных реестров «какие ключи работают».
2. **LLM получает принципы, а не таблицы.** Морфология, числовые эквиваленты, билингвальность формулируются как правила; конкретные пары — это уже work для самой модели на основе `schema.values[]`.
3. **Деградация — по факту, а не по предположению.** Если фильтр не дал результата — сбрасываем его и говорим об этом, а не угадываем заранее.
4. **Lexicon = онтология, не словарь синонимов.** Стартует пустым, растёт через ревью реальных провалов.
5. **Никаких «единственных санкционированных исключений» из инвариантов.** Инвариант либо есть, либо его нет.

После апрува — внедряю всё за один проход.
