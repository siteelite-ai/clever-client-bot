## ⚡ Edge Functions Stability Auditor

### Текущее состояние микро-LLM

Классификатор `classifyProductName()` — ключевой элемент «быстрого пути». При его падении весь title-first поиск пропускается. Сейчас:

1. **Ошибка при API-вызове → `return null`** (строка 689). Нет retry на другой провайдер.
2. **Auto-режим**: выбирает ОДИН провайдер на старте и идёт только к нему. Если Google ключей нет → пробует OpenRouter с `google/gemini-2.5-flash-lite:free` (404) → всё, провал.
3. **Таймаут 3 секунды** — адекватный, но при ошибке нет fallback.

### Риск в плане

**Замена моделей в списке (бесплатные → платные)** — правильный шаг, но **НЕ устраняет single-point-of-failure**. Если платная модель OpenRouter вернёт 500/429 — классификатор снова `null`.

### Рекомендация

Добавить **каскадный retry внутри `classifyProductName()`**: если первый вызов вернул ошибку (не 200) — попробовать следующий провайдер из цепочки. Сейчас этого нет: ошибка = `return null`.

```text
Текущая логика:
  auto → выбрать провайдер → вызвать → ошибка → return null

Нужная логика:
  auto → Google → ошибка? → OpenRouter → ошибка? → Gateway → ошибка? → return null
```

## 💰 Sales Logic Auditor

### Текущее состояние вывода товаров

Блок `articleShortCircuit && productContext` (строка 3204) содержит инструкцию «покажи сразу», но **без жёсткого запрета на уточняющие вопросы** и **без учёта количества** найденных товаров. LLM свободна интерпретировать — и выбирает «уточнить», теряя продажу.

### Оценка плана (≤7 показать / >7 уточнить)

**Хороший порог**, но нужно уточнить:
- Откуда брать число товаров? Из `foundProducts.length`. Это уже есть в коде.
- Порог 7 — для title-first пути. Для ценовых запросов порог 50 (уже реализован).
- Критично: запрет уточняющих вопросов должен быть **абсолютным** при ≤7 товарах, иначе LLM его проигнорирует.

### Рекомендация

В промпте при ≤7 товарах добавить: `ЗАПРЕЩЕНО задавать уточняющие вопросы. Покажи ВСЕ найденные позиции.`

## 📊 Chat Conversion Auditor

### Влияние на воронку

Кейс «кабель ВВГ 3*1,5» — классический: пользователь знает что хочет, товар есть в каталоге, но бот задаёт вопрос вместо показа. Это **потеря конверсии на последнем шаге**.

### Оценка

План корректно адресует обе причины:
1. Классификатор не сработал → товар не нашёлся по title-first
2. Даже если нашёлся → бот задаёт вопросы вместо показа

### Рекомендация

Добавить **regex-страховку для кабельных обозначений** (`NxM,K`). Это не костыль, а safety net для критически важного товарного сегмента. Если микро-LLM упал, но regex видит `3*1,5` — принудительно запустить title-first.

## 🏗️ Архитектор — Сводный вердикт

### Что сейчас есть в системе микро-LLM

| Функция | Что делает | Где вызывается |
|---|---|---|
| `classifyProductName()` | Интент-анализ: название товара, ценовой интент, категория, замена | строка 2774 |
| `resolveSlotRefinement()` | Связывает уточнение с активным слотом | строка 2786 |
| `detectPendingPriceIntent()` | Legacy fallback для клиентов без слотов | строка 2799 |

Классификатор — **единственный вход** в быстрый путь. Его отказ = деградация до общего поиска, который теряет сечение/артикул.

### Оценка плана

| Пункт плана | Вердикт | Комментарий |
|---|---|---|
| Заменить бесплатные модели на платные Gemini | ✅ Правильно | Бесплатные дают 404, платные стабильнее |
| Auto-fallback: заменить `:free` на платную | ✅ Правильно | Но недостаточно — нужен каскадный retry |
| Правило ≤7/>7 товаров | ✅ Правильно | Нужен жёсткий запрет, не мягкий |

### Выявленные риски и как их закрыть

**Риск 1: Single-point-of-failure в классификаторе**
- Сейчас: ошибка API → `return null` → весь быстрый путь мёртв
- Решение: каскадный retry (Google → OpenRouter → Gateway) при ошибке, независимо от выбранного провайдера
- Объём: ~15 строк в `classifyProductName()`

**Риск 2: Промпт недостаточно строгий**
- Сейчас: «покажи сразу» — LLM может проигнорировать
- Решение: добавить явный порог и запрет: `if (foundProducts.length <= 7)` → в промпт `ЗАПРЕЩЕНО задавать уточняющие вопросы`
- Объём: ~10 строк в блоке articleShortCircuit

**Риск 3: Полный отказ всех LLM**
- Regex-страховка для кабельных обозначений (`\d+[*хx×]\d+[.,]?\d*`)
- При детекции → принудительный title-first без микро-LLM
- Объём: ~15 строк перед блоком классификации
