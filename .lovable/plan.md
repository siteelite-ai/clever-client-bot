

# План: полная схема опций категории → корректный матчинг модификаторов

## Что ты сказала, что мы делаем

> «Если нашли категорию — смотрим её опции и работаем с ними. LLM должна понимать, что такое "двухгнёздное", найти эту опцию, забрать все её значения и выбрать правильное.»

Точно так. Для этого LLM должна получить **полную** схему опций категории (все ключи + все значения), а не агрегированную из случайных 30 товаров.

## Корень проблемы (точно)

В `category-first` (стр. 3709–3754):
1. Для каждой найденной подкатегории делается запрос `per_page=30` → получаем `matcherProducts` (30–90 товаров).
2. В `resolveFiltersWithLLM` (стр. 2353–2363) из этих товаров строится `optionIndex` — union опций по выборке.
3. Если в эти 30 товаров не попала ни одна двойная розетка — ключа `kolichestvo_postov` (или как он там называется в API 220volt) **в схеме нет**. LLM физически не может сматчить «двухгнёздная».

То есть мы **не работаем с опциями категории**, мы работаем с **опциями случайной выборки 30 товаров**. Это и есть источник промахов.

## Решение: полная схема опций категории + кэш

### Этап 1 — Classifier (без изменений)
`category="розетка"`, `modifiers=["белая","двухгнёздная"]`. Стр. 736–829 не трогаем.

### Этап 2 — Полная схема опций (новое)

Новая функция `getCategoryOptionsSchema(categoryPagetitle, apiToken)`:

- **Источник схемы**: постраничная агрегация по `?category=<pagetitle>&per_page=200` со страницы 1 до полного покрытия (или max 5 страниц = 1000 товаров). Делается **один раз на категорию**, далее кэшируется.
- **Сбор**: union всех `options[].key` + union всех `options[].value` по каждому ключу. Применяется тот же `isExcludedOption` фильтр, что сейчас.
- **Кэш**: модульный `Map<pagetitle, {schema, ts}>` с TTL 30 мин (по аналогии с `chatCategoriesCache`). Опции категории меняются редко — реальная стоимость одна на категорию в полчаса.
- **Параллелизм**: для нескольких matched-подкатегорий — `Promise.all` по каждой, результат — union схем (одна логическая «крупная категория» = объединённая схема её подкатегорий).
- **Fallback**: если запрос упал/timeout — возвращаем пустую схему, дальше старая логика (агрегация по `matcherProducts`) как safety net. Без регрессий.

### Этап 2b — Использование полной схемы в `resolveFiltersWithLLM`

`resolveFiltersWithLLM` получает новый опциональный параметр `prebuiltSchema?: Map<key, {caption, values}>`:

- Если передан — использует его вместо агрегации по `products`.
- Если не передан — старое поведение (для веток, где полная схема не имеет смысла, например, replacement по конкретному товару).

В `category-first` (стр. 3749–3752) перед вызовом `resolveFiltersWithLLM` подгружаем `getCategoryOptionsSchema(matches, apiToken)` и передаём как `prebuiltSchema`. LLM видит **все** ключи категории и **все** значения каждого ключа.

### Этап 3 — Точный API-запрос (без изменений)

Стр. 3759–3766 уже корректно передаёт `options[key]=value` через `searchProductsByCandidate`. После исправления этапа 2 в `mResolved` будет правильный ключ количества постов → API вернёт точную выдачу.

### Дополнение: убираем literal-query для unresolved

Стр. 3758: `queryText = mUnresolved.join(' ')` — это анти-паттерн. Если LLM не смогла сматчить модификатор с **полной** схемой — значит свойства реально нет в опциях категории, и literal query всё равно даст мусор или ноль. Меняем: если `mUnresolved.length > 0` и `mResolved` непуст — `queryText = null` (доверяем точным фильтрам). Если `mResolved` пуст — единственная попытка с `query=mUnresolved.join(' ')` как сейчас.

## Что НЕ трогаем

- Classifier
- CategoryMatcher (`matchCategoriesWithLLM`)
- Replacement-ветка, бренды, цвет, мощность — резолвятся через тот же путь
- Виджет (предыдущий фикс idempotency)
- `searchProductsByCandidate` — без изменений
- Никаких новых LLM-вызовов, никаких словарей
