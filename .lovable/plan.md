

# Ответ: дело не в количестве чёрных розеток

## Что показывают логи

```
[Chat] Category-first STAGE 2: server options={"cvet__tүsі":"чёрный//қара"}, query="черная двухместная"
[Search] query="черная двухместная" → 0 results
[Chat] STAGE 2 fallback to bucket-N: "Розетки" (priority=2)
```

API получил **противоречивый запрос**: фильтр «чёрный» в `options[...]` И те же слова «черная двухместная» в literal `query`. API сматчил по обоим условиям одновременно → 0 товаров. После этого сработал fallback в bucket «Розетки», который вернул чуть-чуть карточек, и LLM показала 2 случайные чёрные двухместные розетки из этого подмножества.

**Чёрных двухместных розеток в каталоге явно больше двух.** Это не ограничение каталога, а ровно тот же баг suppress-query, который мы уже «починили» на прошлом шаге.

## Почему фикс не сработал

Я проверил логи: ни одной строки `[SuppressQuery] path=bucket-N ...` за весь день нет. При этом в коде helper стоит и вызывается перед формированием Stage 2 (строки 4318–4324 в `chat-consultant/index.ts`).

Вывод однозначный: **helper в коде есть, но edge function chat-consultant работает на старой версии — последний deploy не подхватил изменения**. Поэтому в проде по-прежнему уходит `query="черная двухместная"` вместе с `options[cvet]`, API возвращает 0, и мы видим обрезанные 2 карточки от fallback-ветки.

## Что нужно сделать

1. **Передеплоить edge function `chat-consultant`** — это снимет проблему сразу, потому что helper и все 4 точки его вызова уже в коде.
2. **Проверить логи после деплоя** — должна появиться строка вида:
   ```
   [SuppressQuery] path=bucket-N before="черная двухместная" after="" dropped=[черная, двухместная] → null (allowEmptyQuery=true)
   [Chat] Category-first STAGE 2: server options={"cvet__tүsі":"чёрный//қара"}, query="null"
   ```
3. **Регресс на том же запросе** «Есть черная двухместная розетка?» — ожидание: API вернёт полный набор чёрных двойных розеток (>2 штук), LLM покажет top-3 с приглашением уточнить.

Если после передеплоя `[SuppressQuery]` логи всё равно не появятся — значит вызов идёт по другой ветке (не bucket-N), и нужно будет смотреть, какая ветка реально отрабатывает на этом запросе. Но первый шаг — деплой.
