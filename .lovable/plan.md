

## Аудит фичи «Остатки по складам» — 4 роли

Для этой фичи релевантны 4 роли из AUDIT_PROMPT.md. Вот их анализ и рекомендации.

---

### 1. ⚡ Edge Functions Stability Auditor

**Статус:** Требуется изменение в `chat-consultant/index.ts`

**Находки:**
- Интерфейс `Product` (строка 398) **не содержит поле `warehouses`** — API возвращает его, но TypeScript-тип его не описывает → данные приходят, но не типизированы
- Форматирование контекста (строка 1594) использует только `p.amount` — `warehouses` полностью игнорируется
- `search-products/index.ts` пробрасывает `data.results` as-is — warehouses уже приходят, ничего менять не нужно

**Риски:**
- Массив `warehouses` может быть пустым у некоторых товаров → нужна проверка `if (p.warehouses?.length > 0)`
- Названия городов могут меняться в API → не хардкодить, брать `warehouse.name` динамически

**Рекомендация:** Безопасное изменение. Fallback на `p.amount` если `warehouses` отсутствует.

---

### 2. 💰 Sales Logic Auditor

**Находки:**
- Строка 2438 в промпте LLM 2 **явно запрещает** боту отвечать про склады: «предложи уточнить наличие в нужном филиале у менеджера»
- Это **упущенная конверсия**: клиент хочет узнать наличие в своём городе, а бот отправляет к менеджеру — лишний шаг в воронке

**Рекомендация (лучший вариант):**
Показывать остатки по городам **проактивно**, но только если клиент спрашивает. Формат:
```
Наличие: Актобе — 81 шт., Алматы — 100 шт.
В каком городе вам удобнее забрать?
```
Это сокращает путь к покупке и удерживает клиента в чате.

---

### 3. 📊 Chat Conversion Auditor

**Находки:**
- Текущий путь: вопрос → «не знаю про склады» → «обратитесь к менеджеру» = **3 шага и потеря клиента**
- Оптимальный путь: вопрос → «есть в Алматы 100 шт., в Актобе 81 шт.» → «В каком городе забрать?» = **2 шага и удержание**

**Риск «информационного шума»:**
Если у товара 10+ складов — вывод всех городов перегрузит ответ.

**Рекомендация:** Показывать **только города с amount > 0**, максимум 5-6 городов. Если больше — «и ещё в N городах. Подсказать по конкретному?»

---

### 4. 🎨 Widget UX Auditor

**Находки:**
- Длинный список городов в markdown будет нормально рендериться (просто текст)
- Интеграция с геолокацией (`geoContext` уже есть, строка 2420) — можно **подсвечивать город клиента** первым

**Рекомендация:** Если город определён по гео → показывать его первым: «В вашем городе (Алматы) — 100 шт. Также: Актобе — 81 шт.»

## Итоговый план реализации (консенсус всех ролей)

**Файл: `supabase/functions/chat-consultant/index.ts`**

### Изменение 1: Добавить `warehouses` в интерфейс Product (строка ~418)
```typescript
warehouses?: Array<{
  name: string;
  amount: number;
}>;
```

### Изменение 2: Обновить форматирование контекста (строка ~1594)
Заменить строку с `p.amount` на логику:
- Если `p.warehouses` есть и не пуст → отфильтровать города с `amount > 0` → вывести «Остатки по городам: Алматы: 100 шт., Актобе: 81 шт.»
- Если складов с остатком > 5 → показать первые 5 + «и ещё в N городах»
- Если все склады = 0 или `warehouses` отсутствует → fallback на `p.amount > 0 ? Да (X шт.) : Под заказ`

### Изменение 3: Обновить промпт LLM 2 (строка ~2438)
Заменить правило 6 «Остатки и склады» на:
> «В данных о товарах указаны остатки по городам (поле «Остатки по городам»). Если клиент спрашивает о наличии — показывай данные по конкретным городам. Если клиент упоминает конкретный город — дай точное количество для этого города. Если город определён по геолокации — начни с него. Города с нулевым остатком не упоминай без запроса. Если в данных нет разбивки по городам — покажи общий остаток.»

### Изменение 4: Обновить article-first промпт (строка ~2165)
В блоке `articleShortCircuit` добавить в инструкцию: «Покажи наличие по городам, если данные есть».
