

# Анализ: почему «Гармония» и «черная двухместная розетка» не работают

## Диагностика по логам

### «черная двухместная розетка»
Лог прямо показывает:
```
classify: category="розетка двухместная", modifiers=["черная"]
Category-first: 0 products (24 queries for "розетка двухместная")
```
**Причина**: Классификатор включил «двухместная» в category вместо modifiers. API не находит текст «розетка двухместная» — товары называются «Розетка 2-местная», «РС12-2-ККм двухместн.» и т.д. Если бы category="розетка" и modifiers=["двухместная", "черная"], Pass 1 нашёл бы розетки, а resolveFiltersWithLLM определил бы фильтр по количеству мест и цвету.

### «розетки из коллекции Гармония»
Логов Гармонии нет (ротировались), но из скриншота видно — бот нашёл «Крышки» вместо розеток. Вероятнее всего классификатор поставил `has_product_name=true` (решив, что «розетки из коллекции Гармония» — конкретное название), и запрос пошёл через **title-first** вместо category-first. Title-first искал «розетка Гармония» буквально → нашёл крышки/рамки серии Гармония.

## Корневая проблема: промпт классификатора

Текущий промпт (строка 592) говорит:
> «ВКЛЮЧАЙ все слова, определяющие ПОДТИП товара: количество мест/постов, тип монтажа...»

Это заставляет LLM включать «двухместная» в category. Но API 220volt.kz не поддерживает такой точный текстовый поиск — он работает лучше с короткими общими запросами + фильтрами через options.

Теперь, когда у нас есть `resolveFiltersWithLLM`, нет смысла загружать category подтипами. Чем проще category, тем больше товаров найдёт Pass 1, тем богаче схема характеристик для LLM-резолвера, тем точнее фильтрация в Pass 2.

## План изменений

### 1. Переписать правила category vs modifiers в промпте классификатора
**Файл**: `supabase/functions/chat-consultant/index.ts`, строки 592-598

Новый принцип: category = максимально общий тип товара (1-2 слова), все уточнения (количество мест, тип монтажа, цвет, серия, коллекция, IP, материал, размер) → modifiers.

Было:
- «двухместная розетка» → category="розетка двухместная"
- Логика: «двухместная определяет подтип → в category»

Станет:
- «двухместная розетка» → category="розетка", modifiers=["двухместная"]
- Логика: «category = базовый тип для текстового поиска API; всё остальное в modifiers для фильтрации через характеристики»

### 2. Усилить правило has_product_name для коллекций/серий
Сейчас «розетки из коллекции Гармония» может быть интерпретировано как конкретное название. Нужно добавить в промпт: указание серии/коллекции/линейки — это НЕ конкретное название товара, а фильтр внутри категории.

### Конкретные правки промпта (строки 584-600)

Пункт 4 (product_category) — переписать:
```
4. product_category (string|null): БАЗОВЫЙ тип товара — максимально общее слово или пара слов, 
определяющая товарную группу для текстового поиска в каталоге. 
НЕ включай количество мест/постов, тип монтажа, конструктивные уточнения, 
серию/коллекцию — всё это выносится в search_modifiers. 
Category должна быть достаточно общей, чтобы API нашёл товары этой группы.
```

Пункт 6 (search_modifiers) — переписать:
```
6. search_modifiers (string[]): ВСЕ уточняющие характеристики из запроса, 
не вошедшие в category: количество мест/постов, тип монтажа (накладной, скрытый), 
цвет, бренд, серия/коллекция, степень защиты IP, материал, размер, 
количественные параметры (длина, сечение, ток). 
Если таких нет — пустой массив.
```

Пункт 1 (has_product_name) — добавить:
```
Указание серии, коллекции, линейки товаров — НЕ конкретное название. 
"розетки из коллекции Гармония" → has_product_name=false, category="розетка", modifiers=["Гармония"].
```

## Что НЕ трогаем
- `resolveFiltersWithLLM` — работает (удлинитель 50м доказал)
- Приоритизацию запросов с модификаторами — работает
- Cap=6 — работает
- Title-first, price-intent, replacement — не затронуты

## Ожидаемый результат

**«черная двухместная розетка»:**
- classify: category="розетка", modifiers=["двухместная", "черная"]
- Pass 1: "розетка" → находит 30+ розеток
- resolveFiltersWithLLM видит schema с «Количество разъемов: 1, 2, 3» и «Цвет: белый, черная, кремовый» → filters={количество_разъемов: "2", цвет: "черная"}
- Pass 2: розетки 2-местные черные

**«розетки из коллекции Гармония»:**
- classify: has_product_name=false, category="розетка", modifiers=["Гармония"]
- Идёт через category-first (не title-first)
- Pass 1: "розетка Гармония", "Гармония розетка", "розетка" → находит розетки
- resolveFiltersWithLLM видит «Коллекция: Кварта, Asfora, ГАРМОНИЯ, Florence» → filters={коллекция: "ГАРМОНИЯ"}
- Pass 2: только розетки серии ГАРМОНИЯ
