

## Анализ: Добавление параметра `article` в API 220volt.kz

### Идея

Попросить разработчика 220volt.kz добавить параметр `article` в API (`/api/products?article=MVA25-1-016-C`), чтобы искать товары напрямую по артикулу, а не через общий `query`.

### Это отличная идея. Вот почему:

1. **Точность 100%** — `article` — уникальный идентификатор, API вернёт ровно 0 или 1 товар. Никакой фильтрации на нашей стороне не нужно.
2. **Скорость** — поиск по индексированному полю `article` на стороне БД быстрее, чем полнотекстовый `query`.
3. **Простота** — не нужен regex-детектор артикулов на нашей стороне, не нужна фильтрация результатов.

### Анализ рисков: неправильное понимание контекста

Главный вопрос: **как отличить артикул от обычного текста?** Вот конкретные риски:

| Риск | Пример | Вероятность | Решение |
|------|--------|-------------|---------|
| Обычное слово принято за артикул | «ВА47» — это артикул или начало фразы «ВА47-29»? | Средняя | Regex-детектор с минимальной длиной 5+ символов и обязательным наличием цифр + букв + дефисов |
| Артикул не распознан | «у вас есть артикул CKK11-012-012-1-K01?» | Низкая | Ключевые слова-триггеры: «артикул», «арт.», «код товара», «SKU» |
| Строка похожа на артикул, но это не он | «IP65» — это характеристика, не артикул | Средняя | Список исключений (IP20, IP44, IP65, IP67, IP68 и т.д.) + проверка: если API по `article` вернул 0 — просто fallback на обычный pipeline |
| Пользователь пишет артикул с ошибкой | «MVA25-1-016C» вместо «MVA25-1-016-C» | Низкая | Ничего страшного — API вернёт 0 результатов → fallback на `query` поиск, который может найти по частичному совпадению |
| Два артикула в одном сообщении | «сравни MVA25-1-016-C и MVA25-1-025-C» | Низкая | Детектор должен находить ВСЕ совпадения, не только первое |

**Главный вывод по рискам**: риск ложного срабатывания **безопасен**, потому что если API по `article=X` вернул 0 товаров — мы просто переключаемся на обычный pipeline. Пользователь ничего не заметит, потеря времени — один лёгкий API-запрос (~100-200мс).

### План реализации (после добавления параметра `article` разработчиком)

**Файл: `supabase/functions/chat-consultant/index.ts`**

1. **Функция `detectArticles(message): string[]`** — regex + ключевые слова («артикул», «арт.», «код»). Исключения для IP-рейтингов и прочих ложных срабатываний.

2. **Article-first short-circuit** (перед вызовом LLM 1, ~строка 1860):
   - Если найдены артикулы → для каждого один запрос `?article=<значение>` к API
   - Если найден 1+ товар → пропускаем LLM 1, сразу формируем контекст и идём в LLM 2
   - Если 0 товаров → обычный pipeline (LLM 1 → поиск → LLM 2)

3. **Обновление `searchProductsByCandidate`** — добавить поддержку параметра `article` в URL params.

4. **Обновление `search-products/index.ts`** — добавить параметр `article` (для консистентности, если виджет тоже будет искать по артикулу).

5. **Обновление промптов**:
   - LLM 1: «Если пользователь указывает артикул — intent="catalog", query=артикул AS IS, без синонимов»
   - LLM 2: «Товар по артикулу показывай сразу, без уточняющих вопросов»

### Что попросить у разработчика 220volt.kz

Добавить в `/api/products` необязательный параметр `article` (string) — точный поиск по полю `article` товара. При наличии параметра `article` остальные фильтры можно игнорировать. Возвращать 0 или 1 результат.
