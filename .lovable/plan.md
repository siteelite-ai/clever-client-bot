

# План: Micro-LLM (AI Candidates) на Flash + правка промпта на сохранение всех модификаторов

## Корень проблемы

В коде сейчас **3 разных AI-вызова в search-флоу**, и все они имеют свою историю:

| Слой | Где | Какая модель | Проблема |
|---|---|---|---|
| **Classify** (Micro-LLM intent) | `classifyProductName` | `settings.classifier_model` → дефолт `gemini-2.5-flash-lite` | OK |
| **AI Candidates** (extract_search_intent) | `generateSearchCandidates` (строка 2019) | `aiConfig.model` → **дефолт `google/gemini-2.5-pro`** | ❌ Pro — медленно и не нужно |
| **FilterLLM** (resolve modifiers) | строка 2698 | hardcoded `google/gemini-2.5-flash` | OK |

`aiConfig.model` — это та же глобальная модель, которой формируется **финальный ответ пользователю**. То есть AI Candidates по историческим причинам подцепился к ней. Вот почему он на Pro.

И именно из-за этого в логе по запросу «чёрная двухместная розетка» AI Candidates вернул `option_filters: {}` — Pro проигнорировал блок про обязательное извлечение характеристик. Не потому что Pro «глупее», а потому что промпт допускает разночтение: цвет/материал/количество там идут общим списком с электротехническими характеристиками (цоколь, IP, сечение), и для модели «двухместная»/«чёрная» выглядят как часть названия товара, а не как фильтр.

## Что меняем (минимально, без новой логики)

### 1. AI Candidates → Flash (одна строка)

В сигнатуре `generateSearchCandidates` (строка 2024) дефолт `aiModel` поменять с глобального на **`google/gemini-2.5-flash`** жёстко, по той же схеме, что уже сделано для FilterLLM:

```ts
// Было (строка 4989):
extractedIntent = await generateSearchCandidates(userMessage, aiConfig.apiKeys, historyForContext, aiConfig.url, aiConfig.model, classification?.product_category);

// Станет: захардкоживаем flash для AI Candidates, не меняя aiConfig
const candidatesModel = 'google/gemini-2.5-flash';
extractedIntent = await generateSearchCandidates(userMessage, aiConfig.apiKeys, historyForContext, aiConfig.url, candidatesModel, classification?.product_category);
```

В лог `[AI Candidates] model=google/gemini-2.5-flash`.

**Почему безопасно:** FilterLLM уже работает на Flash и стабилен. AI Candidates — задача того же класса (извлечь структурированные параметры из короткой фразы), Pro здесь избыточен. Финальный ответ пользователю по-прежнему формируется на той модели, что в настройках (Pro) — её не трогаем.

### 2. Правка промпта: жёстко зафиксировать «любой описывающий признак → option_filters»

Сейчас в промпте (строки 2113–2127) примеры option_filters **только про электротехнику** (страна, цоколь, монтаж, защита, напряжение, сечение, длина, розетки). Для модели это сигнал: «характеристика = только техническое». Цвет, размерность, форма, количество мест в этом списке отсутствуют.

**Что делаем:** в блоке «🔧 ФИЛЬТРЫ ПО ХАРАКТЕРИСТИКАМ» меняем формулировку на принципиальную, БЕЗ конкретных примеров типа «чёрный → cvet», как просила:

```
🔧 ФИЛЬТРЫ ПО ХАРАКТЕРИСТИКАМ (option_filters):
ЛЮБОЙ описывающий признак товара, который пользователь упомянул, ОБЯЗАН попасть в option_filters. 
Описывающий признак — это всё, что отвечает на вопросы «какой?», «сколько?», «из чего?», «где?» применительно к самому товару:
- визуальные признаки (цвет, форма, материал, фактура)
- количественные (число элементов, размер, объём, мощность, длина, сечение)
- функциональные (тип монтажа, степень защиты, наличие/отсутствие функции)
- происхождение (страна, бренд если не вынесен в brand)

Правило: если признак стоит в запросе — значит пользователь хочет ИМЕННО ЭТО. Не отбрасывай его, не считай «украшением» к названию товара. 
Ключ option_filters = краткое русское название признака без пробелов (через подчёркивание). Значение = то, что сказал пользователь, в нормальной форме.

Если пользователь не назвал признак — не выдумывай его и не добавляй в option_filters.
```

И **убираем** из промпта длинный блок «Примеры» со строк 2118–2127 — он сужает мышление модели до перечисленных категорий. Оставляем только принцип и формат.

Аналогично смотрим блок 2140–2151 (usage_context) — там примеры останутся, потому что они про другое (контекст использования → IP-предположение), это работает корректно.

### 3. Логирование для проверки

После вызова AI Candidates уже есть лог `[AI Candidates] Human filter: ${filterName}=${filterValue}`. Добавим однострочный summary-лог сразу после извлечения:

```
[AI Candidates] Filters extracted: { цвет: "чёрный", количество_мест: "2" } (model=flash)
```

— чтобы по логам сразу видно было, забрала ли модель цвет/количество мест.

## Что НЕ трогаем

- Модель финального ответа (`aiConfig.model`) — остаётся как в настройках.
- Classify (Micro-LLM intent) — уже Flash-Lite, OK.
- FilterLLM — уже Flash, OK.
- Rerank, suppress-query, роутер веток, slot-state — без изменений.
- Не добавляем few-shot примеров с конкретными цветами/размерами (как просила).
