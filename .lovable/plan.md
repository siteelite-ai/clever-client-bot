

# План: Усиление Micro-LLM для приоритизации name-first

## Проблема

Когда пользователь присылает конкретный товар ("Паяльник-топор высокомощный, серия ЭПСН, 200Вт, 230В, с деревянной ручкой, пакет REXANT, 12-0292"), классификатор должен сразу сказать `has_product_name=true` и отдать полное название. Вместо этого:
1. JSON обрезается из-за `max_tokens: 150` — не хватает места для длинного `product_name`
2. Промпт недостаточно акцентирует приоритет name-first — LLM может разбить запрос на category + modifiers вместо того, чтобы опознать это как конкретный товар

## Что меняем

### Файл: `supabase/functions/chat-consultant/index.ts`

#### 1. Увеличить `max_tokens`: 150 → 300 (строка 614)

Длинные product_name (30+ слов) + search_modifiers = JSON легко превышает 150 токенов. При 300 токенах flash-lite всё ещё отвечает за < 2с, стоимость +$0.0001 за запрос.

#### 2. Переписать правило `has_product_name` в промпте (строка 594)

Текущее правило слишком строгое и запутанное — LLM не понимает разницу между "конкретный товар" и "категория + характеристики".

Новая формулировка — добавить в начало промпта (перед остальными полями) приоритетное правило:

```
ПРИОРИТЕТ №1 — ОПРЕДЕЛЕНИЕ КОНКРЕТНОГО ТОВАРА:
Если пользователь называет товар так, что его можно найти поиском по названию (бренд + модель, 
или тип + серия + параметры, или полное описание с артикулом) — это КОНКРЕТНЫЙ ТОВАР:
  → has_product_name = true
  → product_name = полное название как поисковый запрос
  → search_modifiers = [] (всё уже в названии)

Примеры КОНКРЕТНЫХ товаров (has_product_name=true):
- "Паяльник REXANT серии ЭПСН 200Вт" → product_name="Паяльник REXANT ЭПСН 200Вт"
- "Кабель ВВГнг 3х2.5" → product_name="ВВГнг 3х2.5"
- "ABB S201 C16" → product_name="ABB S201 C16"

Примеры КАТЕГОРИЙ (has_product_name=false):
- "автоматы на 16 ампер" → category="автомат", modifiers=["16 ампер"]
- "розетки с заземлением" → category="розетка", modifiers=["с заземлением"]
- "подбери светильники для ванной" → category="светильник", modifiers=["для ванной"]

Ключевое отличие: если запрос содержит БРЕНД/СЕРИЮ/МОДЕЛЬ + тип товара — это конкретный товар.
Если только тип товара + характеристики без бренда/модели — это категория.
```

#### 3. Добавить JSON recovery (строка 670)

Страховка на случай, если даже 300 токенов не хватит:
- Попытка "закрыть" обрезанный JSON
- Regex-извлечение `has_product_name` и `product_name` из частичного ответа

## Что НЕ меняем

- Никакой логики в коде кроме промпта и max_tokens
- Title-first, category-first, slot-based, replacement — всё остаётся как есть
- Провайдеры, fallback-каскад, таймауты — без изменений

## Риски

| Компонент | Риск |
|-----------|------|
| Категорийный поиск | Нулевой — если has_product_name=false, всё идёт по старому пути |
| Title-first | Улучшается — больше запросов попадёт в быструю ветку |
| Latency | +0.1-0.3с из-за увеличения max_tokens — незначительно |
| Ложные срабатывания | Минимальный — чёткие примеры в промпте разграничивают name vs category |

## Ожидаемый результат

"Паяльник-топор высокомощный, серия ЭПСН, 200Вт, 230В, REXANT, 12-0292":
- `has_product_name=true`, `product_name="Паяльник-топор ЭПСН 200Вт 230В REXANT 12-0292"`
- Title-first поиск → товар найден за 2-3с вместо 66с
