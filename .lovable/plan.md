
# Архитектура ценового поиска и слотовой памяти (v4.3)

## 1. Определение ценового интента — Микро-LLM классификатор

Каждое сообщение пользователя первым делом проходит через **быстрый Gemini 2.5 Flash Lite** (~1 сек), который извлекает:
- `price_intent`: `"most_expensive"` | `"cheapest"` | `null`
- `product_category`: чистое название категории (без разговорного шума)
- `product_name`: конкретное название товара (если есть)

**Пример:** «давай кемпинговый» → `price_intent: null, product_category: "кемпинговый"`

Классификатор получает последние 4 сообщения истории, чтобы понимать контекст (например, что ранее спрашивали про фонари).

## 2. Dialog Slots — структурированная память интентов

### Зачем
Пользователь может сказать «какой самый дорогой фонарь?», потом уточнять «для охоты», «а для кемпинга?» — и бот должен помнить, что ищем САМЫЙ ДОРОГОЙ, даже через 5 ходов диалога.

### Как работает
- **Слот** — объект с полями: `intent` (price_extreme), `price_dir` (most_expensive/cheapest), `base_category` (фонарь), `refinement` (кемпинговый), `status` (pending/done)
- Слоты передаются между клиентом и сервером через SSE-событие `slot_update`
- Клиент хранит слоты в `sessionStorage` (сохраняются при навигации по сайту)
- Максимум **3 активных слота** одновременно
- Pending-слот автоматически закрывается через **4 хода** без обращения к теме

### Жизненный цикл
1. Пользователь: «какой самый дорогой фонарь?»
2. Сервер создаёт слот: `{intent: "price_extreme", price_dir: "most_expensive", base_category: "фонарь", status: "pending"}`
3. Бот спрашивает: «В категории 67 товаров, уточните тип»
4. Пользователь: «давай кемпинговый»
5. Микро-LLM извлекает: `product_category: "кемпинговый"`
6. `resolveSlotRefinement()` находит pending-слот → склеивает: `"кемпинговый фонарь"` + `price_dir: "most_expensive"`
7. Запускается ценовой поиск с правильным интентом

### Клиент-серверный транспорт (Drain Loop)
Сервер отправляет `slot_update` через SSE **после** `[DONE]`. Клиент (embed.js / ChatWidget) использует **drain loop** — после получения `[DONE]` продолжает читать stream до полного закрытия, чтобы не потерять `slot_update`.

## 3. Ценовой поиск — handlePriceIntent

### Алгоритм
1. **Генерация синонимов** (`generatePriceSynonyms`): «кемпинговый фонарь» → 4-8 вариантов (перестановки слов, синонимы из маппинга)
2. **Probe-запрос**: первый вариант, `per_page=1` → получаем `pagination.total`
3. **Решение**:
   - ≤50 товаров → **собираем все** (до 4 запросов параллельно, `per_page=50`), дедупликация, сортировка по цене
   - \>50 товаров → **просим уточнить** (бот задаёт вопрос с вариантами подкатегорий)
   - 0 товаров → пробуем альтернативные запросы из синонимов

### Отказоустойчивость (три уровня)
1. **Таймаут 15 сек + retry**: если API 220volt.kz не ответил за 15 сек, одна повторная попытка с упрощённым запросом
2. **Сохранение интента при fallthrough**: если `handlePriceIntent` вернул `not_found` (таймаут или пусто), `effectivePriceIntent` НЕ сбрасывается — передаётся в основной AI Candidates pipeline
3. **Серверная сортировка + инструкция LLM**: fallback pipeline сортирует найденные товары по цене и добавляет в промт LLM явную инструкцию: «покажи самый дорогой/дешёвый первым»

## 5. Безопасность

- Серверная валидация типов интентов (whitelist: `price_extreme`, `product_search`)
- Санитизация полей `base_category`/`refinement` (удаление HTML, спецсимволов, лимит 200 символов)
- Максимум 3 слота (защита от переполнения)
- Тайм-аут 4 хода для pending-слотов (автоочистка)
