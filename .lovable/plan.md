

# Вердикт Системного Архитектора: Детекция replacement-интента

## Оценка замечания

**Пользователь абсолютно прав.** Хардкод маркеров ("замена", "аналог", "альтернатива") — это антипаттерн. Пользователь может сказать:
- "что-то типа такого, но подешевле"
- "есть что-то подобное?"
- "а если этого нет, что посоветуете?"
- "такой же, но другого производителя"

Ни одно из этих выражений не содержит слов "замена/аналог". LLM справится с этим значительно лучше.

## Решение: расширить существующий Micro-LLM классификатор

У нас уже есть Micro-LLM классификатор (Gemini Flash Lite, строки 536-619), который определяет `has_product_name`, `price_intent`, `product_category`. Нужно добавить **одно новое поле** — `is_replacement: boolean`.

### Изменение в промпте классификатора

Добавить **Задачу 4** в системный промпт (строка 536):

```
Задача 4: Определи, ищет ли пользователь ЗАМЕНУ или АНАЛОГ конкретного товара.
is_replacement=true если пользователь хочет найти похожий/альтернативный товар.
Примеры:
- "предложи замену для ДКУ-LED-03-100W" → is_replacement=true
- "есть что-то похожее?" → is_replacement=true (в контексте ранее найденного товара)
- "а если этого нет в наличии, что посоветуете?" → is_replacement=true
- "такой же, но другого бренда" → is_replacement=true
- "покажи автомат ABB 32А" → is_replacement=false (обычный запрос)
```

JSON-ответ расширяется: `{"has_product_name": bool, ..., "is_replacement": bool}`

### Изменение в парсинге (строка 604)

Добавить `is_replacement: !!parsed.is_replacement` в объект результата.

### Новая ветка в пайплайне

Если `is_replacement === true` и найден исходный товар:
1. Извлечь характеристики исходного товара (мощность, тип, категория)
2. Сформировать поисковые запросы из этих характеристик
3. Запустить `searchProductsMulti()` с ними
4. Исключить исходный товар из результатов
5. Передать LLM оба набора данных с инструкцией "сравни и предложи аналоги"

Если исходный товар НЕ найден — использовать характеристики из названия (LLM уже распарсил).

## Риски

| Риск | Вероятность | Предотвращение |
|---|---|---|
| Ложное срабатывание: обычный запрос определяется как replacement | Низкая | LLM учитывает контекст диалога — "покажи розетку" ≠ "есть что-то похожее на эту розетку". Примеры в промпте чётко разделяют |
| Ложный пропуск: пользователь просит аналог, но LLM не понимает | Очень низкая | LLM значительно лучше ловит семантику, чем regex. Примеры в промпте покрывают основные паттерны |
| Замедление классификатора из-за дополнительной задачи | Минимальная | Это одно boolean-поле, добавляет ~10 токенов к ответу. Таймаут 3 сек не изменится |
| Двойной поиск увеличивает latency на 2-3 сек | Средняя | Приемлемо — пользователь ждёт осмысленный подбор аналогов, а не мгновенный "не могу" |
| Аналоги совсем не похожи на оригинал | Средняя | Rerank по совпадению ключевых характеристик. Если score ниже порога — честный ответ "точного аналога в каталоге нет" |
