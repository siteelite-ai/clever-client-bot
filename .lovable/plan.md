

# Консилиум по плану «suppress resolved tokens from literal query»

Прогнал план через четыре роли: Backend Architect, Search/Retrieval Engineer, Reliability Engineer, Code Reviewer. Ниже — консолидированный вердикт.

## TL;DR

План **системно правильный**, но в текущей формулировке есть **3 риска**, которые нужно закрыть до имплементации, иначе мы аккуратно починим Bucket-N и так же аккуратно сломаем slot-ветку и fallback. Более простого системного решения нет — точечный фикс только Bucket-N как раз и был бы той самой «несистемностью», от которой мы уходим.

## Что в плане однозначно правильно

1. **Единый helper вместо 4 копий suppress-логики.** Это ровно то, к чему вёл предыдущий архитектурный консилиум: один контракт schema → один resolve → один suppress.
2. **Использование того же 4-символьного стема, что уже введён в `resolveFiltersWithLLM`.** Не плодим вторую морфологию — иначе ветки опять разъедутся.
3. **Не трогаем Micro-LLM, роутер и schema-loader.** Корректная зона ответственности фикса.
4. **Регрессионная матрица по 4 путям, а не по фразам.** Это правильный способ ловить дрейф между ветками.

## Риски, которые надо закрыть в плане

### Риск 1 — «over-suppression» в Bucket-N и Matcher
Если стем resolved-значения совпадёт со стемом значимого слова из названия товара (например, модификатор «двойной» застемится так же, как часть названия серии), мы вычистим из query токен, который реально нужен для текстового поиска. Категория-first это переживёт (фильтры точные), но Replacement / alt-bucket — нет.

**Митигация в плане:**
- Suppress применять **только к токенам, которые Micro-LLM явно вернул как `search_modifiers`**, а не ко всем словам query.
- Стем-сравнение — по паре `(modifier_token, resolved_value)` 1:1, а не «всех со всеми».
- Никогда не выбрасывать токены, которые входят в `has_product_name` / название категории.

### Риск 2 — `query = null` ломает контракт API в части веток
В Bucket-N и Matcher `null` безопасен, потому что есть `options[...]`. Но в **alt-bucket / Replacement** мы можем прийти в ситуацию «resolved=всё, options проставлены, query пустая» — и тогда API уйдёт без literal-сигнала вообще, что для fallback-веток даёт нерелевантную выдачу.

**Митигация в плане:**
- `query = null` разрешать **только** в Bucket-N и CategoryMatcher.
- В Slot refinement оставить текущее поведение (там это уже работает корректно после прошлого фикса).
- В alt-bucket / Replacement — **не подавлять query до пустоты никогда**; если после suppress пусто, оставлять оригинал. Эта ветка по определению «менее уверенная».

### Риск 3 — двуязычные значения `ru//kz`
`resolved_values` приходят как `чёрный//қара`. Если helper не разнесёт это по `//` перед стеммингом, мы будем сравнивать стем русского модификатора со склеенной строкой и ничего не вычистим — то есть фикс молча не сработает в Bucket-N тоже.

**Митигация в плане:**
- В helper явный шаг: `value.split('//').flatMap(half => stem(half))`.
- Логировать `resolvedStems=[...]` отдельно, чтобы в логах было видно, что обе половины учтены.

## Чего в плане не хватает (добавить перед имплементацией)

1. **Явный whitelist веток для `query=null`:** только `bucket-N` и `matcher`. Прописать это в helper как параметр `allowEmptyQuery: boolean`, а не «по месту».
2. **Защита от пустого `search_modifiers`:** если Micro-LLM вернул пустой список модификаторов — helper не должен делать suppress вообще, даже если есть resolved filters (значит, фильтры пришли из старого slot, а не из текущего сообщения).
3. **Инвариант-лог `[SuppressQuery] SKIP reason=…`** для случаев «нет модификаторов», «не та ветка», «после suppress пусто и allowEmptyQuery=false». Без этого в логах будет непонятно, почему suppress «не сработал» — это нас уже кусало.
4. **Тест-кейс на over-suppression:** запрос, где модификатор частично совпадает по стему с названием товара. Должен пройти без потери токена названия.

## Более простые альтернативы — рассматривали, отклоняем

- **«Просто не передавать query в Bucket-N, если есть options»** — ломает кейсы, где модификатор резолвлен только частично; literal-часть всё ещё нужна.
- **«Подавлять прямо внутри `resolveFiltersWithLLM`»** — нарушает single-responsibility этой функции (она про резолв, а не про формирование API-запроса) и снова размажет логику.
- **«Регекс-блэклист слов»** — не масштабируется на категории и языки, ровно тот тип патча, от которого мы отказались по `mem://~user`.

Единый helper + явный whitelist веток — минимально достаточное системное решение.

## Итоговый вердикт

План **одобряем к реализации с тремя поправками**:

1. Suppress только по токенам из `search_modifiers`, не по всему query.
2. `query = null` разрешён только в Bucket-N и Matcher (через явный флаг helper'а).
3. Helper обязан расщеплять `ru//kz` перед стем-сравнением и логировать обе половины.

Плюс добавить инвариант-логи `SKIP reason=…` и тест на over-suppression в регрессионную матрицу.

После этих правок риск задеть другие ветки (slot, alt-bucket, Replacement) — минимальный, потому что в них поведение либо остаётся прежним (slot уже починен), либо явно ограничено флагом (`allowEmptyQuery=false`).

## Что меняем в файле

`supabase/functions/chat-consultant/index.ts`:
- новый helper `suppressResolvedFromQuery(query, resolvedValues, modifierTokens, { allowEmptyQuery })`
- вызов из 4 веток с разными флагами:
  - CategoryMatcher: `allowEmptyQuery: true`
  - Bucket-N Stage 2: `allowEmptyQuery: true`
  - Slot refinement: `allowEmptyQuery: true` (поведение совместимо с текущим)
  - Replacement / alt-bucket: `allowEmptyQuery: false`
- инвариант-логи `[SuppressQuery] path=… before=… after=… dropped=… resolvedStems=… SKIP reason=…`

Если согласна с поправками — на следующем шаге перехожу в default mode и реализую helper + 4 точки вызова + логи.
