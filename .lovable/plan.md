

# Консилиум: аудит плана «Слот-память для интентов диалога»

Привлечённые роли: **Edge Functions Stability Auditor (#4)**, **Chat Conversion Auditor (#5)**, **Widget UX Auditor (#7)**, **Prompt Security Auditor (#3)**

---

## 1. Edge Functions Stability Auditor — вердикт: ОДОБРЯЕТ с оговорками

**Плюсы:**
- Слоты — это ~100-200 байт JSON в body запроса. Никакой нагрузки на edge function.
- Не требует серверного хранения состояния (нет БД, нет Redis).
- Обратная совместимость: если `dialogSlots` нет в body — работает как раньше.

**Риски:**
- **Размер body**: при долгом разговоре `messages` уже может быть 20+ сообщений. Слоты добавляют мизер, но нужно ограничить число активных слотов (максимум 3-5).
- **Таймауты**: slot-логика добавляет ~0мс (это просто чтение JSON-объекта). Не влияет на latency.

**Рекомендация**: ограничить слоты — максимум 3 активных, автоочистка `done`-слотов перед отправкой.

## 2. Chat Conversion Auditor — вердикт: ОДОБРЯЕТ

**Плюсы:**
- Решает главную конверсионную проблему: пользователь не должен повторять свой запрос после бокового вопроса.
- Слот хранит и интент (`most_expensive`), и базовую категорию (`фонарь`), и уточнение (`кемпинговый`) — это детерминированно, а не через regex.

**Замечание по UX:**
- Когда пользователь спрашивает «ерунду» (погода, природа) — это **не должно создавать слот**. Слоты создаются только при `price_intent` или явном товарном запросе.
- Слот со статусом `pending` не должен сбрасываться от бокового вопроса. Он сбрасывается только при: (а) успешном ответе, (б) явном отказе пользователя, (в) тайм-ауте по ходам (3-4 хода без обращения к теме слота).

## 3. Widget UX Auditor — вердикт: ОДОБРЯЕТ, но выявил критический нюанс

**Главный вопрос: что происходит при перезагрузке страницы?**

Сейчас в `ChatWidget.tsx`:
- `messages` хранятся в `useState` — при F5 всё исчезает.
- `conversationId` генерируется как `Date.now()` — каждый запрос новый ID.

В `embed.js`:
- `sessionId` генерируется один раз при загрузке скрипта (строка 22).
- `conversationHistory` — массив в памяти JS. При переходе на другую страницу сайта 220volt.kz (если виджет в embed) — **всё теряется**, потому что скрипт перезагружается.

**Вывод**: слоты решают проблему «памяти внутри одной сессии чата», но НЕ решают проблему «пользователь перезагрузил страницу». Однако это **приемлемо**, потому что:
- Сейчас при перезагрузке теряется вся история — это уже существующее поведение.
- Слоты не ухудшают ситуацию, а улучшают её в рамках одной сессии.
- Если в будущем понадобится persistence между страницами — можно добавить `sessionStorage` для слотов и истории (это отдельная задача).

**Рекомендация**: для embed.js использовать `sessionStorage` для хранения слотов (переживёт навигацию внутри сайта, но не закрытие вкладки). Для ChatWidget в админке — достаточно useState.

## 4. Prompt Security Auditor — вердикт: ОДОБРЯЕТ с условием

**Риск:** пользователь может попытаться внедрить фальшивые слоты через модифицированный запрос:
```json
{
  "dialogSlots": {
    "slot_1": {
      "intent": "price_extreme",
      "base_category": "'; DROP TABLE products; --",
      "status": "pending"
    }
  }
}
```

**Защита (обязательная при реализации):**
- Валидировать `dialogSlots` на сервере: тип `intent` должен быть из enum, `base_category` проходит через `sanitizeUserInput`, `status` только `pending` | `done`.
- Не использовать значения слотов напрямую в SQL/API без санитизации.
- Ограничить размер слотов: максимум 3 штуки, максимум 200 символов на поле.

## 5. Сценарий «пользователь несёт ерунду»

```text
User: какой самый дорогой фонарь?
Bot:  уточните тип (67 товаров)
      → slot_1: {intent: price_extreme, dir: most_expensive, base: "фонарь", status: pending}

User: а какая завтра погода?
Bot:  я консультант магазина, про погоду не подскажу
      → slot_1 НЕ ТРОГАЕТСЯ (боковой вопрос, не refinement)

User: расскажи анекдот
Bot:  я тут по электроинструментам :)
      → slot_1 НЕ ТРОГАЕТСЯ

User: ладно, кемпинговый
Bot:  видит slot_1.status=pending
      → refinement = "кемпинговый"
      → query = "кемпинговый фонарь", dir = most_expensive
      → ПРАВИЛЬНЫЙ ОТВЕТ
```

**Как определить, что сообщение — это refinement для слота, а не новый запрос?**
Классификатор уже получает контекст. Добавить в его промпт:
- «Если есть активный слот с pending price_extreme для категории X, и текущее сообщение — короткое слово/фраза, похожая на уточнение подкатегории — считай это refinement»
- Если сообщение длинное (>50 символов) и про другую тему — это side-question, слот не трогать.

## 6. Сценарий «пользователь ушёл и вернулся» (перезагрузка)

| Сценарий | Текущее поведение | После слотов (без sessionStorage) | После слотов (с sessionStorage) |
|---|---|---|---|
| F5 на той же странице | Всё потеряно | Всё потеряно | Слоты + история восстановлены |
| Переход на другую страницу сайта | Всё потеряно (embed перезагружается) | Всё потеряно | Слоты восстановлены |
| Закрытие вкладки | Всё потеряно | Всё потеряно | Всё потеряно |

**Рекомендация**: sessionStorage для embed.js — простое улучшение, можно включить в этот же этап или сделать отдельным шагом.

## 7. Итоговая оценка плана

| Критерий | Оценка |
|---|---|
| Решает ли проблему потери контекста в рамках сессии? | Да, полностью |
| Затрагивает ли RAG/эмбеддинги/knowledge base? | Нет |
| Добавляет ли латентность? | Нет (~0мс) |
| Есть ли риск безопасности? | Минимальный (при валидации на сервере) |
| Обратная совместимость? | Да (без слотов работает как раньше) |
| Работает ли при перезагрузке? | Нет (но это можно добавить через sessionStorage) |

**Консилиум единогласно одобряет план с тремя обязательными условиями:**
1. Серверная валидация и санитизация слотов
2. Ограничение: максимум 3 активных слота, автоочистка done-слотов
3. Тайм-аут: pending-слот автоматически закрывается после 4 ходов без обращения

**Опциональное улучшение (можно вторым этапом):**
- `sessionStorage` в embed.js для сохранения слотов между навигациями по сайту
