## Контекст

На запросе «есть чёрные розетки на два гнезда?» система выдала перчатки и клеммники, проигнорировав модификаторы цвета и количества постов. Логи показали 4 независимые проблемы — две лежат на уровне промптов, две на уровне алгоритма. План правит всё одной волной без слома существующей маршрутизации (slot → article-first → Classify → price → title → category → replacement → AI Candidates).

---

## 1. AI Candidates — модификаторы всегда попадают в `option_filters` (промпт)

**Файл:** `supabase/functions/chat-consultant/index.ts`, блок `extractionPrompt` (стр. 2062–2205).

**Проблема:** FilterLLM на slot refinement правильно извлекает «чёрный» + «два гнезда», но AI Candidates на первом сообщении возвращает пустой `option_filters`. Расхождение поведения двух промптов на одинаковом инпуте.

**Что меняем:**
- Раздел «🔧 ФИЛЬТРЫ ПО ХАРАКТЕРИСТИКАМ» переписываем как один жёсткий принцип: любой модификатор товара (визуальный, количественный, функциональный, происхождение) обязан попасть в `option_filters`. Без расщепления на «фильтр» и «украшение».
- Удаляем длинные блоки few-shot примеров с usage_context, IP44, лампами E27 и т.п. — Flash на длинном промпте теряет инструкцию. Оставляем определения и принципы.
- Принцип «количественные числительные-прилагательные (одинарный/двойной/двухместный/трёхполюсный) — это количественная характеристика, не часть названия» формулируем явно, без перечисления.
- Убираем повторяющиеся 🚨/⚠️ блоки — это шум для модели.

## 2. CategoryMatcher — приоритет бытовых категорий (промпт)

**Файл:** тот же, функция `matchCategoriesWithLLM` (стр. 297–312).

**Проблема:** на «розетки» система выбирает «Силовые розетки» (CEE-разъёмы), у которых нет чёрных бытовых вариантов на два поста.

**Что меняем — добавляем в промпт Правило 7:**

> Если в каталоге для одного и того же предмета есть категория общего/бытового назначения и категория узко-специализированная (промышленная, силовая, профессиональная, высоких номиналов), выбирай общую/бытовую. Специализированную включай только если в самом запросе пользователя или в контексте предыдущих реплик диалога есть явный признак специализированного применения: упоминание промышленности/производства/цеха, трёхфазной сети, конкретного высокого номинала тока или напряжения, специальных стандартов (CEE, IP67+) или класса профессионального инструмента. Признак должен быть в словах пользователя или истории — не додумывай.

Без хардкод-списка категорий. Принцип, который Flash применит к любой паре «бытовое vs промышленное».

Сигнатура `matchCategoriesWithLLM` уже принимает `settings`; контекст истории в неё не передаётся. Добавляем опциональный параметр `historyContext?: string` — короткая выжимка последних 3 пользовательских реплик (без ассистента). Передаём её в промпт перед `ЗАПРОС ПОЛЬЗОВАТЕЛЯ` блоком «КОНТЕКСТ ДИАЛОГА». На вызывающей стороне (стр. ~4083 и далее) собираем `historyContext` из `conversationHistory`.

## 3. Schema fallback — не звать FilterLLM с пустой схемой (алгоритм)

**Файл:** тот же, `getCategoryOptionsSchema` (стр. 1129–1199) и место вызова FilterLLM в slot refinement.

**Проблема:** если API вернул 0 товаров для категории или упал, `getCategoryOptionsSchema` возвращает пустой `Map`. FilterLLM получает пустую схему и резолвит фильтры вслепую → мусор на выходе.

**Что меняем:**
- В месте вызова FilterLLM (`resolveFiltersWithLLM`): если `schema.size === 0`, **пропускаем** FilterLLM-вызов и переиспользуем фильтры, уже резолвленные на этапе category-first / AI Candidates (`resolved_filters` из открытого slot-а).
- Логируем `[FilterLLM] schema empty for "<category>" → reusing prior resolved_filters`.
- Никаких изменений в самой `getCategoryOptionsSchema` — она и так возвращает пустую Map по контракту, просто проверяем размер у вызывающей стороны.

## 4. Domain guard в rerank — отбрасываем нерелевантные категории (алгоритм)

**Файл:** тот же, `rerankProducts` / `scoreProductMatch` (стр. 1930–1946).

**Проблема:** в выдачу «чёрных розеток» попали чёрные перчатки и клеммники — у них в названии есть слово «чёрный», и rerank их посчитал релевантными. Категория товара при оценке игнорируется.

**Что меняем:**
- В `rerankProducts` принимаем дополнительный параметр `allowedPagetitles: Set<string>` — список категорий, выбранных `CategoryMatcher` для текущего запроса.
- После сортировки по скору **полностью удаляем** товары, чей `parent_pagetitle` (или эквивалент в Product) не входит в `allowedPagetitles`.
- Если `allowedPagetitles` пуст или не передан (старые ветки вызова без category-context) — фильтр не применяется, поведение как сейчас.
- На вызывающей стороне передаём `allowedPagetitles`, собранный из результата `matchCategoriesWithLLM` для текущего запроса.
- Логируем количество отфильтрованных: `[DomainGuard] dropped N items from non-allowed categories: [...]`.

**Что НЕ трогаем (защищаем):**
- Маршрутизация: slot → article-first → Classify → price → title → category → replacement → AI Candidates
- Article-first детекция (включая алфавитно-цифровые токены ≥4 симв.)
- `processPriceIntent`, `effectivePriceIntent`, `price_extreme` slot
- Brands-ветка (`extractBrandsFromProducts`, `brandsContext`)
- Replacement, cascading relax non-critical, broadCandidates, English fallback
- JSON-схема `extract_search_intent`, provider lock на Flash
- Передача брендов в исходной форме пользователя (кириллица/латиница)
- Двухшаговый семантический маппинг в FilterLLM («модификатор → физическое свойство → значение схемы»)

## Регрессионные сценарии после деплоя

| Запрос | Ожидание в логах |
|---|---|
| `есть чёрные розетки на два гнезда?` | `[AI Candidates] Filters extracted: {цвет:чёрный, количество_постов:2}` непустой; `[CategoryMatcher]` → бытовые «Розетки», не «Силовые розетки»; `[DomainGuard]` отбрасывает перчатки/клеммники |
| `розетки промышленные на 32А` | `[CategoryMatcher]` → «Силовые розетки» (есть явный маркер «промышленные» + «32А») |
| `розетки` (без контекста) | `[CategoryMatcher]` → бытовые «Розетки» |
| `розетки` после реплики «у нас цех» | `historyContext` содержит «цех» → `[CategoryMatcher]` → силовые |
| Категория с 0 товарами | `[FilterLLM] schema empty → reusing prior resolved_filters`, FilterLLM не вызывается |
| `чёрная двухместная розетка` (slot уже открыт) | FilterLLM работает как раньше — двухшаговый маппинг не сломан |
| `16093 есть?` | `[ArticleDetect]` → article-first, AI Candidates не вызывается |
| `какие бренды розеток?` | brands-ветка, `extractBrandsFromProducts` |
| `самая дорогая бензопила` | `processPriceIntent` |

При регрессии любого сценария — откатывается только соответствующий блок (промпт или функция).
