

# Аудит плана: Роль «Edge Functions Stability Auditor» + «RAG Quality Auditor»

## Что проверяем

План предлагает заменить кастомный replacement-пайплайн (`extractSearchableTraits` → `searchProductsMulti` → `rerankReplacements`) на переиспользование существующего category-first пайплайна, где модификаторы берутся из характеристик оригинального товара.

---

## Вердикт: ОДОБРЕНО с оговорками

План архитектурно верный — он устраняет дублирование логики и использует уже отлаженный механизм (category-first + `resolveFiltersWithLLM`). Но есть риски, которые нужно учесть при реализации.

---

## Риски и как их закрыть

### Риск 1: Условие `!classification?.is_replacement` на строке 3445

**Проблема**: Category-first пайплайн (строка 3445) явно ИСКЛЮЧАЕТ `is_replacement` из своей работы:
```
!classification?.is_replacement && ...
```
Если мы просто перенаправим replacement в category-first, нужно либо убрать это условие для replacement-кейса, либо вызывать category-first логику напрямую из блока replacement (строки 3601-3656).

**Рекомендация**: Вызывать ту же логику (plural → schema → resolveFilters → STAGE 2) из блока replacement напрямую, НЕ трогая условие на строке 3445. Это безопаснее — не затрагивает обычный category-first путь.

### Риск 2: Товар не в каталоге (extractTraitsFromName)

**Проблема**: Сейчас `extractTraitsFromName` обрабатывает случай, когда оригинал НЕ найден в каталоге (строка 3613-3616). В новом плане эта функция удаляется, но альтернатива не описана чётко. Если пользователь пишет «найди замену XYZ-500W» и товара нет в каталоге — откуда возьмём категорию и модификаторы?

**Рекомендация**: Для этого кейса извлекать категорию и характеристики из `classification.product_name` через микро-LLM (тот же классификатор уже умеет определять category и modifiers). Фактически — повторный вызов classify с product_name как запросом.

### Риск 3: Каскадное ослабление (строки 3520-3561)

**Проблема**: Category-first имеет cascading fallback (drop filters one by one). Для замены это правильное поведение — если нет точного аналога 100Вт IP67, система попробует 100Вт без IP67. Этот механизм автоматически переиспользуется — **это плюс плана**.

**Вердикт**: Нет риска, наоборот — улучшение.

### Риск 4: Порог >7 товаров и слоты

**Проблема**: Category-first создаёт `product_search` слот при >7 результатах (строка 3575). Для replacement-кейса это тоже подходит — система предложит уточнить характеристики. Но нужно добавить в слот пометку `isReplacement: true` и ссылку на оригинал, чтобы LLM в финальном ответе сказал «Вот аналоги для X» а не просто «Вот товары».

**Рекомендация**: Сохранять `replacementMeta` как сейчас + передавать его в слот.

### Риск 5: Влияние на другие пути поиска

**Проверка по чек-листу:**
- Article-first (строки ~3350-3380) — НЕ затрагивается
- Title-first (строки 3424-3442) — НЕ затрагивается (replacement сначала ищет оригинал через title-first, потом переходит к аналогам)
- Category-first (строки 3444-3599) — НЕ затрагивается (условие `!is_replacement` остаётся)
- Price intent (строки 3340-3422) — НЕ затрагивается
- RAG / Knowledge Base — НЕ затрагивается (отдельная ветка)

**Вердикт**: Изоляция обеспечена. Другие пути поиска не пострадают.

## Итог аудита

| Аспект | Статус |
|--------|--------|
| Архитектурная корректность | Одобрено — устраняет дублирование |
| Влияние на category-first | Безопасно — если вызывать логику напрямую, не менять условие 3445 |
| Влияние на другие пути | Безопасно — article/title/price/RAG не затронуты |
| Кейс «товар не в каталоге» | Требует доработки — нужен fallback через classify |
| Cascading fallback | Автоматически переиспользуется — плюс |
| Слоты и уточнение | Нужна пометка `isReplacement` в слоте |
| Удаление старого кода | Одобрено — `extractSearchableTraits`, `extractTraitsFromName`, `rerankReplacements` больше не нужны |

**Общий вердикт: План одобрен. Можно реализовывать с учётом рисков 1, 2 и 4.**
