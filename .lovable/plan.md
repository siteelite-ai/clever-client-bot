

# Точечная правка: bucket-фильтр по приоритету категории

## Что не меняем (уже работает правильно как ты описал)

В `chat-consultant`, строки 3530–3556, **уже** делается ровно то, что ты предлагаешь:

1. **Запрос по категории** (`category=Розетки`) — точный поиск
2. **Запрос по названию** (`query=розетка`) — поиск по слову в названии/описании
3. Результаты мёржатся, дедуплицируются по id
4. Группируются по фактической категории каждого товара (`p.category.pagetitle`) — `categoryDistribution`
5. Через `prioritizeBuckets` каждой категории присваивается приоритет: **2** если её имя содержит корень классификатора («Розетки скрытой установки» содержит «розетк» → 2), **0** если не содержит («Колодки клеммные» → 0)

То есть пайплайн уже **знает**, какие категории релевантные («Розетки …»), а какие — мусор от полнотекстового поиска («Колодки», «Удлинители»).

## Что сломано (одно место)

Строки 3585–3625, цикл выбора winner-bucket. Он перебирает все bucket'ы подряд и выбирает тот, у кого больше резолвнутых модификаторов, **игнорируя `bucketPriority`**. Из-за этого «Колодки» (priority=0) могут выиграть у «Розеток» (priority=2), если в колодках случайно отрезолвилось больше фильтров. Дальше поиск уходит в категорию колодок → 0 двойных розеток.

## Минимальная правка

Перед циклом резолва — отсечь bucket'ы с `priority < 2`. Если такие есть хоть один — конкурируют только они. Если ни одного — fallback на старое поведение (значит классификатор ошибся или релевантной категории в выдаче API не оказалось).

```ts
const allBuckets = sortedBuckets.slice(0, MAX_BUCKETS_TO_CHECK);
const relevantBuckets = allBuckets.filter(([name]) => bucketPriority[name] === 2);
const bucketsToTry = relevantBuckets.length > 0 ? relevantBuckets : allBuckets;

    : `[Chat] Category-first: NO buckets match classifier="${effectiveCategory}", fallback to all ${allBuckets.length}`
);

for (const [catName, count] of bucketsToTry) {
  // ... существующий цикл резолва без изменений
}
```

То же самое симметрично — в replacement-блоке (строки ~3868–3940), где есть `replBucketPriority`, но он тоже не используется при выборе.

## Логика в твоих терминах

- Сделали два запроса (категория + название) — как ты сказал ✓ (уже есть)
- Если в выдаче есть категория, чьё имя содержит слово запроса («Розетки …») — она получает приоритет ✓ (уже есть в `bucketPriority=2`)
- При выборе подкатегории сравниваем **только релевантные** между собой — фильтр выбирает лучшую розеточную подкатегорию (скрытой установки vs накладные) ← **новое**
- Если ни одна категория в выдаче не содержит слово запроса — берём по товарам, как раньше (fallback) ✓

Колодки физически не могут больше попасть в winner — у них `priority=0`, они отсеиваются на входе в цикл.
